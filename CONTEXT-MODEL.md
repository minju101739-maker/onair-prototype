# ON AIR 맥락 기반 스케줄링 모델 — 상태등 & 3단계 퍼널

> 이 기획의 **중점**: 회의 조율을 "빈 시간 교집합 문제"가 아니라 **"캘린더 상태등 + 맥락(context) 만족 문제"** 로 재정의한다.
> 기존 캘린더는 *언제 비어 있는지*는 알지만, *그 시간이 나에게 어떻게 느껴지는지*는 모른다.

- 문서 버전: v0.5
- 상위 문서: [CONCEPT.md](./CONCEPT.md)
- 변경 이력: v0.2 — 집계식(5.2)에 **최소 유효 N 단서 + 임계치 N≥3 확정** 반영. / v0.3 — Q1 부분 해소(초대 역할 기반 등급 도출 — FLOW.md Q2) 반영. / v0.4 — 제품 표기 용어를 **내 기준**으로 통일하고, 당일 예외 맥락을 **오늘 상태**로 모델에 추가. / v0.5 — ON AIR 상태등 모델 추가: ON AIR=하드 차단, 피함=소프트 후순위, 비어 있음=파란 기본 후보.

## 1. 중점: "기록되지 않는 맥락"이 문제다

ON AIR의 화면은 모든 시간을 먼저 세 상태로 읽는다:

| 상태 | 모델 의미 | 퍼널 연결 |
|---|---|---|
| **ON AIR** | 회의·업무 일정이 켜진 시간 | ② 제약 조건에서 하드 차단 |
| **피함** | 내 기준·오늘 상태상 피하고 싶은 시간 | ③ 기호 점수에서 감점 |
| **비어 있음** | 일정이 없는 파란 시간 | 후보 생성의 기본 공간 |

| 종류 | 예시 | 기존 앱 | 본 시스템 |
|------|------|---------|-----------|
| 기록 가능한 사실 | "이 날 외근", "이 시간 회의" | ✅ 캘린더 이벤트 | ✅ |
| **기록되지 않는 기준** | "점심 직후는 피하고 싶다", "금요일은 칼퇴하고 싶다", "오늘은 업무가 너무 많다" | ❌ 담을 곳이 없음 | ✅ **내 기준으로 구조화** |

핵심 통찰: 기존 도구는 **이산적 사건(event)** 만 표현한다. 그런데 사람이 회의 시간에 대해 느끼는 불편함의 대부분은 **반복되는 패턴(standing rhythm)** 과 **당일 상태(today context)** 에서 온다. 이걸 데이터로 만들면, "비어는 있지만 잡기 싫은 시간"을 시스템이 이해한다.

## 2. 3단계 퍼널 정의

```
모든 가능한 슬롯
      │
 ① 우선순위 (Priority)      ── "누가 이 슬롯을 죽일 수 있는가?"
      │   필참자 집합만으로 후보 생성. 선택 참석자는 슬롯을 제거하지 못함.
      ▼
 ② 제약 조건 (Constraint)   ── "이 슬롯이 물리적으로 가능한가?" (Yes/No, 하드 필터)
      │   필참자 중 한 명이라도 불가 → 슬롯 탈락.
      ▼
 ③ 기호 점수 (Preference)   ── "가능한 슬롯 중 어느 것이 가장 덜 불편한가?" (랭킹, 소프트)
      │   탈락 아닌 감점. 점수 합산·공정성으로 정렬.
      ▼
   최적 슬롯 (Top-N)
```

핵심 구분:
- **②는 필터(생사 결정), ③은 랭킹(우열 결정).** 이걸 섞으면 "약한 불편함" 때문에 슬롯이 사라지거나, "절대 불가"가 점수로 희석되는 오류가 난다.
- **①은 ②/③의 적용 범위를 정한다.** 선택 참석자의 외근/내 기준은 슬롯을 죽이지 않고 약하게만 반영.

### 2.1 퍼널의 목적 — 선택지 공간의 붕괴 (Option-Space Collapse)

이 퍼널의 가치는 "최적 슬롯을 찾는 것"보다 **선택지를 확실한 소수로 붕괴시키는 것**에 있다. 기존 캘린더앱은 ②까지만 해서 "되는 시간이 너무 많은" 상태로 끝나고, 사용자는 거대한 투표 매트릭스에 던져진다. 우리는 ③까지 적용해 **선택의 폭을 좁히되 확실하게** 만든다.

**예시: 6인 회의 (필참 4 + 선택 2), 2주 탐색**

| 단계 | 적용 | 남는 후보 | 비고 |
|------|------|-----------|------|
| 0. 탐색 범위 | 평일 10일 × 30분 단위 업무시간 | **~160 슬롯** | "12날짜×10시간 투표"의 정체 |
| ① 우선순위 | **필참 4명** 교집합 = 하한선. 선택 2명은 슬롯을 죽이지 않고 4/6→6/6 **커버리지로 등급화** | 범위 한정 + 등급 부여 | 과잉 제약 방지 |
| ② 제약(하드) | 4명 free/busy ∩ 룸 가용 ∩ 이동가능 | **~40 슬롯** | ← **여기까지가 기존 캘린더앱** |
| ③ 기호(소프트) | 각자 불편한 시간 감점·컷 + 선택 2명 약하게 반영 | **상위 3~5개** | ← **우리 서비스의 차별점** |

→ 사용자는 **120칸을 투표하는 대신 3~5개 중 고른다.** 선택폭은 줄지만 확실성은 오른다.

> **Q3 해소:** 이 붕괴 과정에서 "누가 어느 시간을 막았는지"는 **노출할 필요가 없다.** 출력은 살아남은 소수 슬롯뿐이고, 붕괴의 사유(제약·내 기준)는 각자의 개인 평가기 안에 머문다. 주최자는 "왜/누가"가 아니라 **"확실한 3~5개"** 만 받는다.

## 3. 각 단계 구체화

### ① 우선순위 — "나는 필참이 아닌디요?"
- 참석 등급: `필참(required)` / `선택(optional)` / `참고(FYI, 비참석·기록만)`.
- 등급은 **본인이 스스로 태깅**하거나 주최자가 지정 → 충돌 시 협상 규칙 필요(열린 질문 Q1).
- 효과 — 선택 참석자는 **버리지 않고 슬롯의 등급을 올린다(upgrade)**:
  - 후보 슬롯은 **필참자 교집합 = 하한선(floor)** 에서 생성. 4/6 필참이 모두 되면 그 슬롯은 일단 **유효(투표 가능)**.
  - 각 슬롯에 **참석 커버리지**를 표시: 같은 시간에 선택 참석자도 되면 4/6 → 5/6 → 6/6 으로 **색상/등급 상승**. (선택자는 슬롯을 *죽이지* 않고 *밝힌다*.)
  - 따라서 **4/6 슬롯도, 6/6 슬롯도 둘 다 투표 후보**. 동일 조건이면 커버리지 높은 슬롯을 우대(③ 점수에 가산).
  - → 과잉 제약(over-constraint)으로 "되는 시간이 없음" 문제를 구조적으로 완화하면서, "이왕이면 다 모이는 시간"도 놓치지 않음.

> **시각화:** 결과는 가부 이분법이 아니라 **커버리지 히트맵**. 예) 진한 색 = 6/6 전원, 옅은 색 = 4/6 필참만. 주최자는 "전원 모이는 진한 슬롯"과 "필참만 되는 옅은 슬롯"을 보고 고른다. (누가 빠지는지 신원은 비노출 — [DATA-SOURCING 9](./DATA-SOURCING.md))

### ② 제약 조건 — 하드, Yes/No
- 예: "이 날 외근", "이 시간 회의 있음".
- **중요: 캘린더에 없는 제약도 포함한다.** 외근은 캘린더상 "한가"로 보일 수 있음(=출근만 안 할 뿐). free/busy만으로는 못 잡으므로 **별도 제약 입력**이 필요.
- 제약 출처 2종:
  - **연동 캘린더 free/busy**: 이벤트 디테일 없이 바쁨 구간만.
  - **수동/규칙 제약**: 외근, "이 요일은 종일 불가" 등 캘린더에 없는 것.
- 규칙: 필참자 중 1명이라도 위반 → 슬롯 **탈락**.

### ③ 기호 점수 — 소프트, 핵심 (4장에서 깊게)
- 예: "퇴근 전에는 하기 싫다", "점심시간엔 하기 싫다".
- 탈락이 아니라 **감점**. 모든 후보가 살아남고, 점수로 줄 세움.
- 이 레이어가 본 시스템의 차별점 전부를 담는다.

## 4. 기호(내 기준)의 구조화 — 기준을 데이터로

제품 표기는 **내 기준**으로 통일한다. 내부 모델명은 기존 호환성을 위해 `PreferenceRule`을 쓸 수 있지만, 사용자 화면에는 선호 프로필·선호 규칙·내 리듬 같은 표현을 쓰지 않는다.

### 4.1 내 기준 스키마
```jsonc
PreferenceRule {
  when:     <시간 술어>,          // 언제 적용되나
  polarity: "avoid" | "prefer",   // 피함 / 좋음
  weight:   0..3,                 // 강도 (0 없음, 1 약함, 2 보통, 3 강함)
  scope:    "always" | "this_week" | "today" | { until: date }  // 상시 vs 한시적
}
```

### 4.2 `when`의 앵커(anchor) 4종 — "맥락성"의 핵심
내 기준은 **절대 시계**가 아니라 **개인 리듬과 오늘 상태**에 걸린다. 같은 "오후 5시"라도 칼퇴족과 야근족, 오늘 컨디션이 낮은 사람에게 의미가 다르다.

| 앵커 종류 | 표현 | 예시 |
|-----------|------|------|
| 절대 시간 | `14:00–15:00` | 특정 시간대 회피 |
| 요일 | `금요일` | "금요일은 칼퇴" |
| **개인 리듬 앵커** | `퇴근_전(120분)`, `점심_직후(60분)`, `출근_직후(30분)` | 본인 근무시간 기준 상대 시간 |
| 상대 위치 | `다른_회의_직후`, `집중블록_침범` | 회의 사이 버퍼, 딥워크 보호 |
| 오늘 상태 | `컨디션_낮음`, `업무량_많음`, `회의피로_높음` | 오늘만 후보 후순위 |

> 개인 리듬 앵커가 차별의 핵심: 사용자가 **근무시간(출근/점심/퇴근)만 한 번 설정**하면, "퇴근 전", "점심 직후"가 자동으로 실제 시각으로 환산된다. → 입력 부담 최소화 + 맥락 정확도 ↑.

### 4.3 내 기준 카테고리 택소노미 (구체 예시)
1. **시간대 리듬**: 오전 집중형/오후 집중형, 점심 직후 60분 회피.
2. **요일 리듬**: 월요일 오전 회피(주간 정리), 금요일 오후 강한 회피(칼퇴/마무리).
3. **경계 시간**: 퇴근 전 2시간 회피, 출근 후 30분 워밍업 회피.
4. **부하 리듬**: 하루 회의 N개 초과 시 감점, 연속 회의 회피, 회의 간 15분 버퍼.
5. **맥락 전환 비용**: 딥워크 블록을 쪼개는 위치 강한 회피.

6. **오늘 상태**: 오늘 컨디션 낮음, 오늘 업무량 많음, 오늘 회의 피로 높음.

### 4.4 예시 (사용자 A의 내 기준)
```jsonc
근무시간: { 출근: 10:00, 점심: 12:30–13:30, 퇴근: 19:00 }
기준:
  - { when: 점심_직후(60분),   polarity: avoid, weight: 2, scope: always }
  - { when: 금요일 ∧ 퇴근_전(120분), polarity: avoid, weight: 3, scope: always }
  - { when: 오전(출근~점심),    polarity: avoid, weight: 1, scope: always }  // 오후 집중형
```
이 모든 값은 **본인만 소유·열람**(상위 문서의 위임-집계 모델). 동료는 A의 점수도 사유도 못 본다.

## 5. 점수 계산 모델

### 5.1 개인 슬롯 점수 (사용자 u, 후보 슬롯 s)
1. **하드 체크**: u의 제약(②) 위반 시 → u가 필참이면 슬롯 탈락, 선택이면 큰 고정 감점.
2. **소프트 페널티**:
   ```
   penalty(s, u) = Σ_r [ weight(r) × overlap(s, r.window) × polaritySign(r) ]
   ```
   - `overlap` = 슬롯이 규칙 적용 구간과 겹치는 비율(0~1). 30분 회의가 점심직후 60분 구간에 30분 다 걸치면 1.0.
   - `prefer`(좋음)는 음수(=보너스), `avoid`(피함)는 양수(=벌점).

### 5.2 팀 집계 — 누구의 불편을 줄일 것인가
| 방식 | 식 | 성격 | 위험 |
|------|-----|------|------|
| 합산(공리주의) | `Σ_u penalty(s,u)` 최소화 | 전체 불편 최소 | 특정인이 매번 희생 |
| **최소최대(평등주의)** | `max_u penalty(s,u)` 최소화 | 최악 경험자 보호 | 평균이 약간 나빠짐 |
| **권장: 혼합 + 공정성** | `α·합산 + β·최소최대 + γ·누적공정성` | 균형 | 가중치 튜닝 필요 |

> ⚠️ **최소 유효 N (집계 익명성의 전제):** 위 집계가 개인 점수를 가려주는 것은 **신호가 섞일 사람이 충분할 때뿐**이다. 참석자 N이 작으면 익명성이 떨어지고, **N=2에선 0이 된다** — 내 점수를 빼면 상대 점수가 그대로 분리되는 차분 공격이 성립. **결정: 이 집계-익명 모드는 N≥3에서만 적용하고, N≤2는 별도의 대칭적 상호 공개 모드로 분기한다** (정의 → [CONCEPT 4.2](./CONCEPT.md), 배경 → [CONCEPT R4](./CONCEPT.md)).

- 선택 참석자는 가중치 절반(예: ×0.5) 등으로 약화.
- **참석 커버리지 가산**: 슬롯에 추가로 참석 가능한 선택자가 많을수록 점수 보너스(4/6 < 5/6 < 6/6). 단, 필참 페널티를 뒤집을 만큼 크면 안 됨 — 커버리지는 동급 슬롯 간 타이브레이커 수준.
- **누적 공정성(γ)**: 과거에 "희생 슬롯"을 떠안은 사람일수록 이번엔 보호 → 시차 팀의 새벽 회의 같은 부담을 로테이션.

## 6. 내 기준 캡처 — 묻지 않고 데이터에서 채운다 (Derive)

> 최대 리스크였던 "아무도 폼을 안 채운다"는 **입력 방식을 Ask→Derive로 바꿔 해소**한다.
> 사용자가 작성하는 게 아니라, 시스템이 데이터를 긁어 채우고 사용자는 **확인(승인)만** 한다. 상세 → [DATA-SOURCING.md](./DATA-SOURCING.md)

1. **데이터 자동 채움**: 캘린더 free/busy + 회의실 예약 기록 + 과거 회의 행동 로그에서 제약·내 기준을 추론.
2. **앵커 자동화**: 근무시간(출근/점심/퇴근)을 HR/근태 데이터로 채워 상대 앵커(`퇴근_전`, `점심_직후`) 자동 환산.
3. **확인 루프**: "데이터상 금요일 오후 회의를 자주 옮기던데, 피함 기준으로 둘까요?" — 작성이 아니라 승인.
4. **자연어 보정(선택)**: 사용자가 원하면 "금요일은 일찍 끝내고 싶어"를 기준으로 변환해 추가 (개인 평가기 내부 LLM).
5. **오늘 상태 반영**: "오늘 컨디션이 낮다", "오늘 업무가 너무 많다"는 `scope:"today"` 기준으로 만들고 하루가 지나면 자동 만료한다.

## 7. 워크스루 — 30분 회의, A 참석

후보 두 개를 A의 내 기준(4.4)으로 점수화:

| 후보 슬롯 | 하드 | 소프트 페널티 계산 | 점수 |
|-----------|------|--------------------|------|
| 목 11:00 (오전) | 통과 | 오전 회피 weight 1 × overlap 1.0 = 1.0 | **+1.0** |
| 금 17:30 (퇴근 전) | 통과 | 금∧퇴근전2h weight 3 × overlap 1.0 = 3.0 | **+3.0** |
| 목 14:00 (점심직후 지남) | 통과 | 13:30 종료라 점심직후 구간(13:30까지) 안 겹침 = 0 | **0.0** |

→ A에게는 **목 14:00**이 최선. 팀 집계 시 다른 참석자 점수와 합산·공정성 보정으로 최종 결정. **A가 왜 금요일을 싫어하는지는 누구에게도 노출되지 않음.**

## 8. 프라이버시 연결

- 이 퍼널 전체(①②③)는 상위 문서의 **개인 평가기(Personal Evaluator)** 안에서 돈다.
