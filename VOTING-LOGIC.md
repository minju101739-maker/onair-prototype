# ON AIR 수렴 로직 — 투표가 아니라 "제안 → 동의/비동의"

> 이 시스템의 수렴(언제로 정할지)은 **투표가 아니다.** 개인 평가기가 이미 내 기준을 반영해 점수를 냈으므로,
> 사람이 또 시간을 고르는 것은 중복이고, 내 기준을 노출시키며, [거대한 투표 매트릭스](./CONTEXT-MODEL.md#21-퍼널의-목적--선택지-공간의-붕괴-option-space-collapse)를 되살린다.
> 올바른 모델: **ON AIR가 파란 비어 있음 중 가장 좋은 시간을 제시 → 참석자는 시간을 고르지 않고 동의/비동의(이분법)만.**

- 문서 버전: v0.2
- 변경 이력: v0.2 — **동의는 회의 생성 후 프로세스**로 확정: 위저드는 1순위 발송(가예약+동의 요청+상신)까지, §4의 동의/재제안/확정은 홈 "동의 현황"에서 진행.

## 1. 왜 "투표"가 아니라 "동의/비동의"인가

| | 투표(구) | 동의/비동의(채택) |
|---|---|---|
| 사용자 행위 | 여러 시간 중 **좋은 시간을 골라** 투표 | 제시된 슬롯에 **확인·거절**만 |
| 내 기준 노출 | 어디에 투표했는지 = 기준이 드러날 여지 | 가/부만 — 내 기준은 애초에 개인 평가기 안 |
| 인지 부하 | N개 시간 × 비교 = 매트릭스 | 1개 제안에 O/X |
| 결정력 | 표 분산 → 애매 | 필참 전원 동의 = 즉시 확정 |
| 중복 | 엔진이 점수 냈는데 사람이 또 고름 | 엔진=내 기준, 사람=확정 (역할 분리) |

**핵심:** 내 기준은 **후보를 만드는 단계**(개인 평가기, 3단계 퍼널)에서 이미 반영된다. 사람이 하는 마지막 행위는 **"그 시간에 실제로 참석 가능한가"의 확정(commit)** 이지 기준 표명이 아니다. 그래서 이분법이 맞다.

> 예외 없음: 상급자 포함(수직) 모드도 동일하게 동료는 동의/비동의만 하고, 그 결과를 상급자에게 **보고·승인**받는다([FLOW.md §7.3](./FLOW.md)).

## 2. 명확한 후보시간군 — 순위화 (1·2·3순위)

"되는 시간을 다 띄우는" 대신, 엔진이 **순위가 매겨진 소수 후보**를 낸다.

```
① 우선순위(필참 교집합) → ② 제약(하드 필터) → ③ 기호(소프트 감점)
                                          ▼
                              점수 정렬 → 상위 1·2·3순위
```
- 정렬 키: **필참 커버리지(내림) → 기준 감점(오름) → 전체 커버리지(내림) → 이른 날짜/시간**.
- 노출은 상위 **3개**까지. 주최자·참석자는 120칸을 보지 않고 "확실한 1·2·3순위"를 본다.
- 각 후보에 등급(S/A/B)과 커버리지(필참 n/m)를 표기.

---

## 3. 최악의 조건 — 모두가 만족할 시간이 없을 때 ⚠️

이게 이 문서의 핵심. **"전원 만족" 슬롯이 없어도 화면은 비지 않는다.** 품질을 3등급으로 낮춰가며 **항상 명확한 후보를 낸다:**

| 품질 | 조건 | 화면 | 후보 산출 |
|------|------|------|-----------|
| **good** | 전원 가능 + 기준 통과 | 배너 없음 | 편한 순 상위 3 |
| **tight** | 전원 가능하나 **아무도 안 편함**(기준 컷 전멸) | ⚠️ "모두에게 편한 시간이 없습니다. 가능한 시간 중 제안합니다." | 감점 적은 순 상위 3 |
| **partial** | **전원 가능한 시간이 아예 없음** | ⚠️ "전원 가능한 시간이 없습니다. 가장 많은 인원이 가능한 시간을 제안 — 범위 확대·온라인 전환·필참↔선택 조정 검토." | **최다 커버리지** 순 상위 3 (예: 4/5명) |

- partial에서도 각 후보에 **필참 n/m**을 표기해 "누구까지 되는지"(신원 아닌 **수**)를 보여준다.
- 구제책을 함께 제시: 탐색 범위 확대 · 형식 전환(온라인이면 이동 제약 소멸) · 필참을 선택으로 강등 · 소요 단축.
- **프로토타입은 이 최악 경로를 상시 노출**: 홈의 데모 회의를 전원(7명) 참석으로 열면 tight 배너가 바로 뜬다.

## 4. 동의/비동의 진행 & 재제안(폴백) — 회의 생성 "후"의 프로세스

> ⚠️ 이 절의 흐름은 **위저드 밖**에서 일어난다. 주최자의 회의 만들기는 "1순위 발송"으로 끝나고(가예약 상태), 아래 동의·재제안·확정은 참석자 응답에 따라 홈의 **동의 현황**에서 진행된다. 동의는 주최자가 시키는 단계가 아니라 참석자가 하는 행위이기 때문이다.

```
1순위 제안  ──▶  참석자 동의/비동의
                     │
       ┌─────────────┼──────────────┐
   필참 전원 동의    필참 1명 비동의     응답 대기
       │                 │
      확정        1순위 미성립 → "2순위로 재제안"
                          │  (proposalIdx++)
                     2순위 제안 …  (소진 시 "대안 없음 — 범위·참석자 조정")
```

- **성립 조건**: 제안 슬롯에 대해 **필참(상급자 제외) 전원 동의 + 주최자 가용**. FYI·선택은 성립에 영향 없음(선택은 커버리지만).
- **비동의 1명이면 그 슬롯은 미성립** → 시스템이 **다음 순위로 재제안**을 권한다. 사람이 협상 핑퐁을 하는 대신 순위 폴백으로 수렴.
- **비동의 사유는 비공개** — 집계는 동의/비동의 수만. "누가 왜 거절했는지"는 개인 경계 안(P1).
- 후보가 소진되면(모든 순위 비동의) "대안이 없습니다 — 범위·참석자를 조정하세요"로 안내(회의 미성립 = 정상 신호, [FLOW.md Q6](./FLOW.md)).

### 4.1 의사코드
```js
propose(rankedCandidates()[proposalIdx]);
onResponse(user, yes/no):                 // 참석자는 가/부만
  if (필참(비상급자).allYes && organizerFree) → confirm(slot);
  else if (필참.anyNo) → offer(reProposeNext);   // proposalIdx++
report/approve if 상급자 포함;              // 수직 모드(§7.3)
```

## 5. 수직 모드(상급자 포함)와의 결합
- 동료는 위 동의/비동의를 그대로 수행(상급자는 응답 대상 제외).
- 동료 확인이 끝나면 **확인된 슬롯 + 순위 대안**을 상급자에게 보고 → 승인 → 확정.
- 상급자에게 투표를 요구하지 않는다. 동료의 확정 의사가 보고안의 근거가 된다.
