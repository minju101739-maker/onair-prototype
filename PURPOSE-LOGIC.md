# 회의 목적 로직 — 시스템의 중심축

> 이 시스템의 모든 것은 **"왜 이 회의를 하는가(목적)"** 에서 파생된다.
> 목적은 하나가 아니라 **복합적**이며, 목적의 조합이 곧 회의의 구성(누가·어떻게·무엇을 준비)을 결정한다.
> 이 문서는 그 파생 규칙을 스크립트(실행 가능한 규칙)로 정리한다.

- 문서 버전: v0.3
- 변경 이력: v0.3 — 역할 요구를 **필수 → 권장**으로 완화(결정권자는 선택의 영역). 결정권자(상급자)가 명단에 있을 때만 발송 시 상신 체계가 붙는다. §4.1의 하드 검증 폐기 → 권장 안내.
- 작성일: 2026-07-04
- 상위 문서: [CONCEPT.md](./CONCEPT.md) · 관련: [FLOW.md](./FLOW.md) §0-A(현행 목적 표기)·§7.5(복수 목적)

## 1. 원칙 — 목적은 복합적이고, 모든 것을 파생시킨다

1. **목적은 단일하지 않다.** "정보 공유와 결정을 한 자리에서" 처럼 회의는 여러 성격이 섞인다. 목적은 **집합**이다.
2. **목적이 구성을 만든다.** 목적을 고르면 필요한 역할·형식·기록자·소요시간·산출물, 그리고 **각 참석자가 준비할 것**이 자동으로 결정된다. 사용자가 하나하나 정하는 게 아니라 목적에서 도출된다(Ask→Derive).
3. **목적은 간접 전달 장치다.** "당신은 이 회의의 결정권자입니다 — 판단 기준을 준비해 오세요" 처럼, 초대장이 각 사람에게 왜 불렸고 무엇을 준비해야 하는지를 목적·역할로부터 자동 생성해 전한다.

### 1.1 목적을 "고르게" 하지 말고 "안건에서 읽어내라" (v0.2)

초기 UI는 사용자에게 목적을 칩으로 **직접 고르게** 하고, 안건을 한 줄씩 입력·추가하게 했다 — 구조화 노동을 사용자에게 전가한 것으로, Ask→Derive 원칙과 정면으로 어긋난다. 실제 사용자의 회의 메모는 이미 "안건1. Atlas 배포 공유 / 안건2. 통합 방향 결정 …" 처럼 **한 번에 쏟아내는** 형태다.

> **뒤집힌 모델(채택):** 사용자는 안건을 **자유롭게 적거나 붙여넣는다**. 시스템이 (a) 각 줄을 안건으로 구조화하고(머리표 `1.`·`-`·`•` 제거), (b) 각 안건의 **유형을 내용에서 자동 분류**하며, (c) **회의 목적 = 안건 유형의 합집합**으로 파생한다. 목적을 따로 고르는 단계는 사라진다. 각 안건에는 분류된 유형이 **태그로 표시**되고, 태그를 눌러 바로잡을 수 있다.

- 유형 자동 분류: 키워드 사전(결정/공유/발산/업무 조율/관계) 매칭, 못 잡으면 사용자가 선택한 목적 또는 `기타`로 폴백한다. (프로토타입 휴리스틱 — 실서비스에선 LLM 분류로 고도화.)
- 목적은 회의 단위 **파생값**이지 입력값이 아니다 → 舊 Q-F13(안건별 목적 태깅) 채택: 유형은 **안건별**로 갖고, 회의 목적은 그 합집합.

## 2. 목적별 기준표 (Purpose Spec) — 단일 목적의 규칙

역할 택소노미(4종, [FLOW.md Q2](./FLOW.md)): `결정권자(decide)` / `정보제공(info)` / `실행(exec)` / `참고(fyi)`.
> **v0.3 완화:** 역할은 전부 **권장**이다 — 결정권자 없는 결정 회의도 있을 수 있다(주최자 본인이 결정하거나, 결정을 다음으로 미루는 회의). 결정권자가 **있으면** 발송 시 상신(보고) 체계가 자동으로 붙고, 없으면 보고 없이 동의 요청만 나간다.

아래 표의 "필수/권장"은 이 4종 역할 위에 목적별 **의미 라벨**을 입힌 것이다 (같은 info라도 공유 회의에선 "공유자", 정렬 회의에선 "이해관계자").

| 목적 | 필수 역할 (≥1) | 권장 역할 | 형식 정책 | 기록자 | 산출물 | 권장 시간 |
|------|---------------|----------|----------|--------|--------|-----------|
| **결정** decide | — (하드 필수 없음) | **결정권자(decide)**·정보제공(info) | 동기 필수 — 대면·화상 (비동기 불가) | 필수 | 결정 기록 | 30분 |
| **발산** ideate | 참여자 3+ (info·exec) | — | 대면 우대 (화이트보드) | 권장 | 아이디어 목록 | 90분 |
| **공유** inform | — | 공유자(info) | **자유** — 온라인·대면 무관, 비동기 대체 가능 | 선택 (문서로 대체) | 문서 | 30분 |
| **정렬** align | — | 이해관계자(info)·결정권자(decide) | 하이브리드·대면 | 권장 | 합의 사항 | 30분 |
| **관계** bond | — (전원 대등) | — | 대면 | 없음 | 없음 | 60분 |
| **기타** other | — | 참여자(info) | 하이브리드 기본·수정 가능 | 권장 | 회의 결과 | 30분 |

### 2.1 목적별 "각 역할이 준비할 것" (Prep) — 간접 전달의 핵심

| 목적 | 결정권자 | 정보제공 | 실행 | 참고(FYI) |
|------|----------|----------|------|-----------|
| 결정 | 판단 기준·최종 책임 | 결정에 필요한 근거·데이터 | 실행 가능성 검토 | (종결 카드 수신) |
| 발산 | — | 사전 아이디어 스케치 | 사전 아이디어 스케치 | — |
| 공유 | — | **공유 자료 준비** | — | 사전 읽기(자료 있으면) |
| 정렬 | 조율 방향 | 자기 팀 입장·제약 정리 | 실행 일정 확인 | (합의 사항 수신) |
| 관계 | — | — | — | — |

> 이 Prep 문구가 초대장에 실려, 받는 사람이 "나는 왜 불렸고 무엇을 준비하나"를 즉시 안다.

## 3. 복합 목적의 결합 대수 (Combination Algebra)

목적이 집합 `P = {p₁, p₂, …}` 일 때, 구성 파생값은 다음 연산으로 계산한다. **핵심 원칙: 필요는 합집합, 제약은 교집합(가장 엄격한 쪽), 크기는 최댓값.**

| 파생값 | 결합 규칙 | 근거 |
|--------|----------|------|
| **필수 역할** | **합집합(∪)** — 각 목적의 필수 역할을 모두 요구 | 결정+공유 → 결정권자 **AND** 공유자 둘 다 필수 |
| **권장 역할** | 합집합에서 필수와 겹치는 건 제외 | 중복 안내 방지 |
| **형식** | **가장 엄격한 쪽** (동기 필요 우선) | 결정(동기 필수)+공유(자유) → **동기 필수**. 공유가 있다고 비동기로 못 감 |
| **소요 시간** | **max(목적별 권장, 안건수×15분)** | 섞인 회의는 가장 긴 것 이상 필요 |
| **기록자** | 하나라도 "필수/권장"이면 지정 | 결정·정렬·발산 포함 시 기록 필요 |
| **의사결정 방식** | 결정이 포함될 때만 노출 | 결정 없는 회의는 맺음 방식 불필요 |
| **산출물** | 합집합 | 결정 기록 + 문서 등 여러 산출물 |
| **비동기 게이트** | **집합이 {공유}뿐일 때만** 발동 | 결정 등 동기 목적이 섞이면 회의가 정당 → 게이트 없음 |
| **준비 사항** | 역할별로 목적들의 Prep 합집합 | 결정+공유의 정보제공자 → "근거·데이터" + "공유 자료" 둘 다 |

### 3.1 형식 정책 상세 (동기성 랭크)

각 목적에 동기성 랭크를 부여하고, 집합의 **최댓값**이 형식 기본값·잠금 여부를 정한다:

| 목적 | 랭크 | 기본 형식 | 자유도 |
|------|------|-----------|--------|
| 공유 inform | 0 | 온라인 | **자유** (온/오프/비동기 모두 허용) |
| 정렬 align | 1 | 하이브리드 | 동기 권장 |
| 결정 decide | 2 | 대면 (화상 허용) | **동기 필수** (비동기 불가) |
| 발산 ideate | 2 | 대면 | 대면 우대 |
| 관계 bond | 3 | 대면 | 대면 |

- **집합 최대 랭크 = 0** (공유만): 형식 자유 — 사용자가 온/오프 자유롭게, 비동기 전환 게이트 노출.
- **집합 최대 랭크 ≥ 2** (결정·발산·관계 포함): 대면 기본, 비동기 불가. 온라인/하이브리드로 변경은 가능하되 "동기 권장" 안내.
- 규칙은 **잠금이 아니라 기본값 + 안내**다 (UX 스트레스 최소화 — [FLOW.md §7.6](./FLOW.md)). 단 결정 포함 시 비동기 전환만 차단.

## 4. 파생이 각 플로우 단계에 미치는 영향

```
목적 집합 P
   │
   ├─▶ [구성] 필수 역할 = ∪ need(p)  → 참석자 검증("결정권자 1명 필요")
   │         권장 역할 안내 · 형식 기본값(랭크 max) · 기록자 노출 여부 · 의사결정 방식 노출 여부
   │
   ├─▶ [가용성/후보] 소요시간 = max · 형식에 따라 슬롯 정의((시간) vs (회의실×시간))
   │
   ├─▶ [수렴] 비동기 게이트({공유}뿐일 때) · 위계 분기(§7.3)는 참석자 구성에서
   │
   ├─▶ [확정 초대장] 목적 배지 + 안건 + 역할별 준비 사항(Prep) 자동 삽입
   │
   └─▶ [종결] 산출물 종류 = ∪ out(p) → 안건별 결론 카드
```

### 4.1 구성 단계 검증 규칙 (의사코드)

```js
function validateComposition(P, attendees){
  const need = union(P.map(p => SPEC[p].need));       // {role: label}
  for (const [role, label] of need)
    if (!attendees.some(a => a.role === role))
      return `이 회의엔 ${label}가 필요합니다`;          // 하드 검증
  if (P.includes("decide") && !hasDecisionMode())
    return `결정 방식을 정해 주세요`;
  return OK;
}
```

### 4.2 초대장 준비 사항 생성 (의사코드)

```js
function prepLines(P, attendees){
  return distinctRoles(attendees).map(role => {
    const texts = P.flatMap(p => SPEC[p].prep[role] || []);
    return texts.length ? `${roleLabel(role)}: ${unique(texts).join(" · ")}` : null;
  }).filter(Boolean);
}
// 예) 결정+공유 · 참석자에 결정권자·정보제공 있음
//  → "결정권자: 판단 기준·최종 책임"
//     "정보제공: 결정 근거·데이터 · 공유 자료 준비"
```

## 5. 워크스루

### 5.1 문제 발생 → 결정 (스크린샷 사례)
- 계기 `문제 발생` → 목적 기본 `결정`. 안건 2개.
- 필수 역할 = {결정권자}. 형식 = 대면 기본(동기 필수). 기록자 필수. 의사결정 방식 노출.
- 구성 단계: "결정권자 1명 필요" 안내 → 없으면 진행 차단.
- 초대장: 결정권자에게 "판단 기준을 준비", 정보제공에게 "원인 데이터를 준비".

### 5.2 결정 + 공유 (한 번에)
- 목적 = {결정, 공유}.
- 필수 역할 = {결정권자} ∪ {공유자(info)} = 결정권자 **그리고** 정보제공 각각 1명 이상.
- 형식 = max(rank) = 결정(2) → 대면 기본, 비동기 불가. (공유가 있어도 결정 때문에 동기)
- 게이트 = {공유}뿐이 아니므로 **미발동**.
- 소요 = max(30,30)=30, 안건수 반영.
- 초대장: 정보제공자에게 "결정 근거·데이터 · 공유 자료" 둘 다 준비.

### 5.3 정기 리듬 → 공유만 (자유 입력 자동 분류)
- 계기 `정기 리듬`(기본 목적 `정렬`) 선택 후, 안건에 "분기 실적 **공유**" 한 줄 입력.
- 자동 분류: "공유" 키워드 → 안건 유형 `공유`. 회의 목적 = {공유} (파생).
- 필수 역할 = {공유자}. 형식 자유(온라인 기본). 기록자 선택.
- 게이트 = {공유}뿐 → **발동**: "문서로 충분할 수 있습니다".

### 5.4 붙여넣기 한 방 (실제 팀 메모)
입력(붙여넣기):
```
1. Atlas 대시보드 배포 현황 공유
2. Atlas+QA 통합 방향 결정
- QA 사용 방식 가이드 정렬
Jonathan 릴리즈 후 피드백 수집 아이디어
```
→ 자동 구조화: 안건 4개, 유형 `[공유][결정][정렬][발산]`. 회의 목적 = {결정, 발산, 공유, 정렬}(합집합). 필수 역할 = 결정권자 ∪ 공유자 ∪ 이해관계자. 형식 = 대면(결정·발산 랭크 2). 권장 90분(발산). 게이트 미발동. **사용자는 목적을 한 번도 고르지 않았다.**

## 6. 열린 질문
- **Q-P1.** 필수 역할 최소 인원 — 발산 "참여자 3+", 정렬 "이해관계자 2+" 같은 수량 조건을 하드로 강제할지, 권장으로만 둘지.
- **Q-P2.** 형식 "동기 필수(결정)"에서 온라인도 동기이므로 허용 — 그런데 "대면이어야만 하는 결정"(민감 사안)은 어떻게 구분할지.
- **Q-P3.** Prep 문구를 목적 기본값으로만 둘지, 주최자가 안건별로 커스터마이즈하게 할지.
- **Q-P4.** 역할별 준비 사항을 초대 시점(사전)에 더해, 회의 직전 리마인드로도 보낼지.
