# ON AIR 회의 플로우 — "왜?"와 조율에서 기록까지 이어지는 수명주기

> 문제의식: 지금까지의 설계는 **"회의는 이미 하기로 되어 있다"** 를 전제로, 시간·장소 조율(중류)만 정교하게 다뤘다.
> 실제 사용자에게 회의는 조율 이전(**왜 하는가**)에서 시작해 조율 이후(**무엇이 결정됐나**)에서 끝난다.
> ON AIR는 여기에 한 가지 질문을 더한다: **"내 캘린더에서 지금 켜진 시간, 피해야 할 시간, 비어 있는 파란 시간은 어디인가?"**
> 이 문서는 실제로 회의를 구성하는 사람의 관점에서 "왜?"와 시간 상태를 함께 보며 플로우 전체를 다시 도출한다.

## 0-A. 현행 생성·상세 플로우 우선 규칙 (v0.22)

- 사용자 표기는 `계기`가 아니라 **목적**으로 통일한다. 내부 `origin` 필드는 마이그레이션 호환을 위해서만 유지한다.
- 목적 선택은 `결정 필요 / 문제 발생 / 정보 공유 / 정기 회의 / 이전 회의 후속 / 기타` 6종이다. 분류하기 어려운 회의는 `기타`로 시작해 목적과 안건을 직접 적는다.
- 후보 탐색 기간은 실제 시작일·종료일로 지정하며 기본 2주, 최대 4주다. 한 주 요일 칩에 가두지 않는다.
- 발송 전 확인과 받은 동의·참석 요청은 같은 초대장 카드 구조를 쓴다. 안건이 2개 이상이면 첫 안건과 펼침 아이콘을 표시한다. 화살표뿐 아니라 첫 안건 제목 전체에 호버·포커스·클릭을 제공하며, 호버·포커스는 임시 열림, 클릭은 고정 열림으로 동작한다.
- 캘린더 회의·동의 요청·참석 요청·동의 현황·회의 기록의 공통 상세는 하나의 정보 카드로 연다. 순서는 항상 `회의명 → 목적 → 일시 → 장소 → 안건 → 참석자`이며, 상태는 위쪽, 응답·후속 액션은 아래쪽에만 붙인다. 회의 기록은 공통 카드 아래에 결론·액션 결과만 확장한다.
- 사용자 표기 `ON AIR 확정`은 ON AIR에서 조율을 마친 회의라는 출처·상태다. 현재 진행 중이라는 뜻이 아니며 캘린더 카드·범례·목록 배지는 모두 프라이머리 블루로 표시한다.
- `모두에게 편한 시간이 없음`처럼 실패로 읽히는 표현 대신 `참석 가능한 시간 중 선택된 추천안`처럼 선택 결과를 말한다.

이 절은 아래의 과거 `계기(Origin)` 설계 기록보다 우선한다.

## 0. 진단 — 현재 플로우는 어디서 시작하고 어디서 끝나나

```
실제 회의의 수명주기:
  계기 → 목적 → 참석자/안건 → 형식(온/오프) → [시간·장소 조율] → 초대 → 진행 → 기록 → 결정 → 후속 → 회고
                                                    ▲
                                        현재 시스템은 이 구간만 다룸
                                        (index.html: 제목입력 → 참석자 → 가용성 → 후보 → 투표 → 확정)
```

| 실제 사용자의 질문 | 현재 시스템의 답 |
|---|---|
| 왜 이 회의를 해야 하지? | 없음 — 첫 화면이 "회의 제목을 입력하세요" |
| 이 회의 목적이 뭐지? | 없음 — 제목 문자열이 전부 |
| 나는 왜 초대됐지? 안건은? | 없음 — 안건은 비목표(CONCEPT N2), 초대장엔 {제목, 시간, 방}만 |
| 온라인으로 하면 안 되나? | 없음 — 슬롯이 (회의실×시간)으로 오프라인을 암묵 전제 (DATA-SOURCING 4) |
| 이 회의는 언제 누가 하기로 정한 거지? | 없음 — 발의의 출처(provenance) 기록 없음 |
| 회의록은 누가 쓰고, 결정은 어떻게 맺지? | 없음 — 비목표(N2) |
| 필참인데 못 가면 어떻게 되지? | 없음 — 필참/선택은 슬롯 점수 계산에만 쓰임 |

→ 결론: **조율(middle)은 과잉 설계, 발의(upstream)와 종결(downstream)은 공백.** 사용자가 겪는 회의의 고통은 조율에만 있지 않다 — "왜 하는지 모르는 회의에 불려가서, 아무것도 결정되지 않고 끝나는 것"이 더 크다. G4(불필요한 회의 감축)를 진지하게 믿는다면, 그 판단은 시간 조율 단계가 아니라 **발의 단계**에서 이루어져야 한다.

## 1. 질문 사슬 (Why-Chain) — 각 질문이 플로우에 강제하는 것

### Q0. 왜 회의를 하는가? — 계기(Origin)

회의는 무(無)에서 생기지 않는다. 반드시 계기가 있다:

| 계기 유형 | 예시 | 정당성 |
|---|---|---|
| 결정 필요 | "A안/B안 중 골라야 함" | 강함 — 동기적 합의가 필요 |
| 문제 발생 | "장애 원인 논의" | 강함 — 긴급·상호작용 필요 |
| 정보 공유 | "분기 실적 공유" | **약함 — 대부분 비동기로 대체 가능** |
| 정기 리듬 | "주간 싱크" | 관성 — 주기적 재검증 필요 |
| 상위 지시 / 이전 회의의 후속 | "지난 회의에서 하기로 함" | 출처 기록 필요 |

**플로우에 강제하는 것:**
- 발의 시 **계기를 선택**하게 한다 (자유 텍스트가 아니라 유형 선택 — 데이터가 됨).
- 계기가 "정보 공유"이고 결정 사항이 없으면 → **비동기 역제안 관문**이 여기서 발동한다. G4는 P1+ 확장 기능이 아니라 **플로우의 첫 번째 게이트**다.
- 계기가 "정기 리듬"이면 → N회 반복 후 "이 정기 회의, 아직 필요한가요?" 재검증 루프의 대상이 된다.

### Q1. 회의의 목적은 무엇인가? — 목적(Purpose)이 나머지 전부를 결정한다

목적 유형은 5개면 충분하다: **결정(decide) / 발산(ideate) / 공유(inform) / 정렬(align) / 관계(bond)**.

목적이 정해지면 아래가 **도출**된다 (사용자가 일일이 정하는 게 아니라 기본값이 잡힌다):

| 목적 | 적정 인원 | 적정 길이 | 형식 기본값 | 산출물 |
|---|---|---|---|---|
| 결정 | 결정권자 + 정보보유자 (소수) | 30분 | 동기 필수 | **결정 기록** |
| 발산 | 4–8명 | 60분 | 대면 우대 | 아이디어 목록 |
| 공유 | 제한 없음 | — | **비동기 우선 (문서/녹화)** | 문서 |
| 정렬 | 이해관계자 | 30분 | 동기/하이브리드 | 합의 사항 |
| 관계 | 팀 | — | 대면 | 없음 (그래도 됨) |

**플로우에 강제하는 것:**
- 목적 유형 선택 = 발의의 필수 입력 *(v0.7에서 복수 선택으로 확장 — [§7.5](#75-목적은-복수다-v07). 실제 회의는 공유+결정처럼 유형이 섞임)*. 목적이 "공유" 단독인데 동기 회의를 고집하면 시스템이 이유를 묻는다.
- 이후 단계(참석자 규모, 길이 기본값, 형식, 기록 방식)가 목적에서 자동 제안된다 — Ask가 아니라 Derive([DATA-SOURCING 1](./DATA-SOURCING.md))의 원칙을 발의 단계에도 적용.

### Q2. 누가 참여해야 하고, 무엇이 안건인가?

"누구를 부를까"는 이름 검색의 문제가 아니라 **역할 도출**의 문제다. 결정 회의라면: 이 결정의 **결정권자**는 누구인가, 판단에 필요한 **정보를 가진 사람**은 누구인가, 결과를 **실행할 사람**은 누구인가, **알기만 하면 되는 사람**은 누구인가.

**플로우에 강제하는 것:**
- 참석자 추가 시 이름과 함께 **초대 사유(역할)** 를 붙인다: `결정권자 / 정보제공 / 실행담당 / FYI`. 이것이 기존의 필참/선택/FYI 등급([CONTEXT-MODEL 3-①](./CONTEXT-MODEL.md))의 **근거**가 된다 — 지금은 등급만 있고 "왜 그 등급인지"가 없다.
  - 결정권자·정보제공 → 필참 기본값. 실행담당 → 선택 기본값. FYI → 비참석·회의록 수신.
  - CONTEXT-MODEL Q1(등급을 본인이 정하나 주최자가 정하나)의 부분 해소: **주최자는 역할을 부여하고, 본인은 이의를 제기**한다. "나는 이 결정의 결정권자가 아닌데요"는 "나는 필참이 아닌데요"보다 판정 가능한 문장이다.
- **안건 최소 1줄 필수** *(v0.3에서 안건 목록으로 확장 — [§7.2](#72-회의는-복수-안건-컨테이너다))*. 안건을 쓸 수 없다면 그 회의는 목적이 없는 것이다. 비목표 N2("회의록·안건은 범위 밖")는 이 지점에서 **부분 철회**되어야 한다 — 안건 *문서 도구*를 만들자는 게 아니라, 안건 *한 줄의 존재*가 발의 관문이라는 뜻.
- 초대받은 사람의 화면에는 **"당신은 왜 초대되었는가"** 가 표시된다: `역할: 정보제공 — 안건 2번 항목 관련`.

### Q3. 어디서 하는가? 온라인인가 오프라인인가?

현재 설계는 슬롯을 `(회의실 × 시간)`으로 정의해 **오프라인을 암묵 전제**한다. 실제로는 형식이 조율보다 먼저다 — 형식이 탐색 공간 자체를 바꾼다:

| 형식 | 슬롯 정의 | 하드 제약 |
|---|---|---|
| 온라인 | `(시간)` | 이동시간 제약 소멸, 회의실 불필요 |
| 오프라인 | `(회의실 × 시간)` | 룸 가용·정원·이동시간 (기존 설계 그대로) |
| 하이브리드 | `(회의실 × 시간)` + 원격 참여자 | 현장 참석자에게만 이동 제약 적용 |

**플로우에 강제하는 것:**
- 형식은 목적·참석자 위치에서 **제안**된다: 참석자가 지점 분산이면 온라인/하이브리드 기본값, 발산·관계 목적이면 대면 우대. (참석자 위치 신호는 회의실 예약 데이터에서 이미 추론 가능 — [DATA-SOURCING 2](./DATA-SOURCING.md). 단 N≥3 익명 원칙 유지: "3명이 타지점"이라는 집계만, 누가인지는 비노출.)
- 조율 엔진 입장에서는 **탐색 공간 파라미터**가 하나 추가될 뿐이다. 퍼널(①②③)은 그대로 동작한다.

### Q4. 이 회의는 언제, 어디서 하기로 결정된 것인가? — 출처(Provenance)

초대받은 사람의 흔한 경험: 캘린더에 갑자기 꽂힌 회의, 맥락 없음, "이게 뭐지?" → 주최자에게 DM → 수작업 핑퐁. CONCEPT 1.1이 비판한 "메신저 왕복"은 시간 조율에서만 생기는 게 아니라 **맥락 부재**에서도 생긴다.

**플로우에 강제하는 것:**
- 발의에 **출처 필드**: `이전 회의 X의 후속 / 슬랙 스레드 / 이슈 #123 / 직접 발의`. 정기 회의면 "매주 반복, 최초 발의 6/30".
- 확정 통보(초대장)가 `{제목, 시간, 방}`에서 **`{제목, 시간, 장소/링크, 목적, 안건 번호 목록, 출처, 나의 역할}`** 로 확장된다. 이것은 개인 프라이버시 데이터가 아니라 **회의의 공유 속성**이므로 P1(최소 노출)과 충돌하지 않는다 — 오히려 회의 축 데이터는 참석자 전원에게 투명한 것이 원칙이다 (§4 참고).

### Q5. 회의록은 누가 작성하며, 의사결정은 어떻게 맺는가?

산출물 없는 회의는 목적 달성 여부를 검증할 수 없고, 검증할 수 없으면 G4(회의 감축)의 근거 데이터도 영영 생기지 않는다. 회의록 *도구*(전문 노트·녹화·전사)는 여전히 비목표로 두되, **최소 종결 프로토콜**은 플로우의 일부여야 한다:

- **기록 책임은 내부 파생**: 종결 카드를 작성한 사용자를 시스템이 감사 정보로 보존하되, 회의 생성에서 별도 기록자를 묻거나 조회 화면에 노출하지 않는다.
- **결정 방식 선언**: 목적이 "결정"이면 발의 때 맺음 방식을 고른다 — `책임자 단독 결정 / 다수결 / 전원 합의`. 회의가 산으로 가는 원인의 태반은 "어떻게 끝나는지"를 아무도 정하지 않은 것.
- **종결 카드 (회의 직후 1분)**: `안건별 결론 1줄(완료/이월) + 액션 아이템(담당·기한) + 다음 회의 필요 여부` *(v0.4에서 안건 단위로 확장 — [§8](#8-회의-기록-적재-구조-v04--다름을-같음으로))*. 이 카드가 FYI 등급 참석자에게 배포되는 바로 그 산출물이다.

### Q6. 참석하지 못하면 불이익이 있는가? — 불이익이 아니라 성립 조건으로 재정의

"불참 시 페널티"는 감시 도구로 흐른다 (CONCEPT N3: 근태 관리 도구가 아니다 — 유지). 올바른 재정의는:

> **필참자의 불참 = 개인의 잘못이 아니라, 회의 성립 조건의 붕괴.**

- 필참자(결정권자·정보제공)가 수락하지 않으면 회의는 **미성립** 상태 — 시스템이 자동으로 재조율 또는 연기를 제안한다. 결정권자 없는 결정 회의를 강행하는 것 자체가 낭비이므로.
- 선택 참석자의 불참은 무불이익. 종결 카드(Q5)를 받는 것으로 충분 — **"참석 안 해도 정보에서 소외되지 않는다"** 가 보장되어야 선택 참석자가 부담 없이 빠질 수 있고, 그래야 회의가 작아진다. 회의 감축(G4)의 숨은 전제.
- 불참 이유는 묻지 않는다 (프라이버시 모델 그대로 — 가부만, 사유는 개인 경계 안).

### Q7. 회의가 끝난 뒤에는? — 피드백 루프

종결 카드에 참석자 익명 원탭 피드백을 붙인다: **"이 회의, 필요했나요?" (필요 / 짧아도 됐음 / 비동기로 충분했음)**.

- 집계는 N≥3 익명 원칙 그대로 (개인 응답 비노출, 주최자에게는 분포만).
- 이 데이터가 [DATA-SOURCING 3](./DATA-SOURCING.md)의 "과거 회의 행동 로그"에 **의도 신호**를 추가한다 — 리스케줄 패턴(간접 추론)보다 훨씬 강한 신호로, 다음 발의 때 역제안의 근거가 된다: "이 정기 회의는 최근 3회 연속 '비동기로 충분' 60%였습니다."

## 2. 재구성된 플로우 — 6단계 수명주기

```
 Phase 0. 발의 (Initiate)        ── 왜 하는가?
 │  계기 선택(→ 안건 질문·목적 기본값 결정) → 안건 목록(1개 이상) → 출처
 │  ▶ 게이트: 목적=공유 ∧ 결정 없음 → "문서/녹화로 충분합니다" 역제안 (G4)
 │  ▶ 게이트 통과 실패 시 여기서 끝 — 회의 없이 비동기 산출물 생성. 이것도 성공 경로다.
 ▼
 Phase 1. 구성 (Compose)         ── 누가, 어떤 형식으로?
 │  역할 기반 참석자(결정권자/정보/실행/FYI → 필참·선택 자동 도출)
 │  형식(온라인/오프라인/하이브리드 — 위치·목적에서 제안)
 │  결정 방식 선언(결정 회의인 경우) · 기록 책임은 종결 카드 작성자에서 자동 파생
 ▼
 Phase 2. 조율 (Coordinate)      ── 언제, 어디서?  ★ 기존 시스템 전체가 이 단계
 │  3단계 퍼널(①우선순위 ②제약 ③기호) · 위임-집계 프라이버시 모델
 │  슬롯 = 형식에 따라 (시간) 또는 (회의실×시간)
 │  N≥3 익명 집계 / N≤2 대칭 공개 모드 분기 — 모두 그대로 유지
 │  수렴: 순위화 1·2·3순위 → 동의/비동의(이분법) → (상급자 포함 시) 보고·승인 (VOTING-LOGIC.md · §7.3)
 ▼
 Phase 3. 확정 (Commit)          ── 성립했는가?
 │  초대장 = {시간, 장소/링크, 목적, 안건 번호 목록, 출처, 나의 역할}
 │  필참자 전원 수락(+수직 모드: 상급자 승인) = 성립 조건. 미성립 시 자동 재조율/연기 제안
 ▼
 Phase 4. 진행·종결 (Conduct)    ── 무엇이 결정됐나?  (도구 최소주의)
 │  종결 카드: 안건별 결론 1줄(완료/이월) + 액션(담당·기한) + 다음 회의 필요 여부 (§8)
 ▼
 Phase 5. 환류 (Learn)           ── 필요한 회의였나?
    종결 카드 배포(FYI 포함) · 익명 피드백("필요했나?")
    → 행동 로그로 환류 → 다음 Phase 0의 역제안 근거
    → 정기 회의는 N회마다 재검증 트리거
```

**핵심 원칙 3가지:**
1. **모든 단계는 "왜?"에 답을 남긴다.** 계기·목적·역할·출처·결정방식은 전부 나중에 참석자가 "이 회의 뭐지?"라고 물을 때의 답이다. 답을 만들 수 없는 회의는 발의 단계에서 걸러진다.
2. **회의 축 데이터는 투명, 개인 축 데이터는 비열람.** 목적·안건·역할·종결 카드(기록)는 참석자 전원 공개 (회의의 공유 속성). 내 기준·오늘 상태·가용성·사유·개인 피드백은 기존 프라이버시 모델대로 개인 경계 안 (§4).
3-1. **위저드는 발송까지다 (v0.15).** "회의 만들기"의 산출물은 성립된 회의가 아니라 **발송된 제안**(추천 시간안 + 대안 + 가예약 + 초대·동의 요청 + 결정권자 있으면 상신)이다. 동의는 참석자들이 회의 생성 **후에** 하는 행위이므로 위저드 안에 있을 수 없다 — 동의 진행·재제안·성립 확정은 홈의 "동의 대기" 카드에서 이어진다.
3-2. **상신 체계는 결정권자가 있을 때만.** 결정권자는 권장이지 필수가 아니다 — 명단에 결정권자(상급자)가 있으면 발송과 함께 상신이 올라가고, 없으면 보고 체계 없이 바로 동의 요청만 나간다.
3. **회의가 안 열리는 것도 성공 경로.** Phase 0 게이트에서 비동기로 전환되면 플로우는 실패한 게 아니라 최상의 결과를 낸 것이다. 성공 지표(CONCEPT 11)의 "비동기 전환율"이 이 경로의 측정치.

## 4. 프라이버시 모델과의 정합성 — 충돌하지 않는다

새로 추가되는 데이터가 CONCEPT의 원칙(P1~P5)과 충돌하는지 점검:

| 새 데이터 | 축 | 공개 범위 | 판정 |
|---|---|---|---|
| 계기·목적·안건·출처 | 회의 축 | 참석자 전원 | P1 무관 — 회의의 공유 속성. 오히려 비대칭 정보(주최자만 맥락을 앎)를 해소 |
| 초대 역할 (결정권자 등) | 회의 축 | 본인 + 주최자 (전원 공개 여부는 Q-F1) | 역할은 직무 사실이지 사적 맥락 아님 |
| 종결 카드 (결정·액션) | 회의 축 | 참석자 + FYI | 회의의 산출물 — 공개가 목적 |
| "필요했나?" 피드백 | **개인 축** | **익명 집계만 (N≥3 원칙 적용)** | 기존 위임-집계 모델과 동일하게 처리 |
| 불참 사유 | 개인 축 | **수집하지 않음** | 가부만 — 기존 원칙 그대로 |

> 요지: **회의 축(what we meet about)은 투명하게, 개인 축(why I can't/won't)은 비열람으로.** 이 구분이 서면 프라이버시 모델은 수정 없이 확장을 수용한다.

## 5. 상위 문서 수정 제안 (CONCEPT.md 반영 대상)

1. **N2 부분 철회**: "회의록·안건 작성은 범위 밖" → "회의록 *도구*(전사·녹화·문서 편집)는 범위 밖. 단 **안건 1줄(발의 관문)과 종결 카드(결정·액션 1줄)는 플로우의 필수 단계**."
2. **G4 격상**: "불필요한 회의를 줄이는 역제안"을 P1+ 확장에서 **P0 발의 게이트**로. 조율이 다 끝난 회의에 "사실 비동기로 되는데요"라고 말하는 것은 너무 늦다.
3. **G 신설 (G6)**: "참석자 전원이 회의의 목적·안건·자신의 역할을 초대 시점에 안다."
4. **MVP 재정의**: MVP = Phase 0(게이트 포함) + 1 + 2 + 3. Phase 4·5(종결 카드·피드백)는 P1이되, 데이터 스키마에는 처음부터 자리를 마련 (`Meeting.purpose, origin, agenda, roles[], decisionMode, closingCard`).
5. **성공 지표 추가**: 발의 게이트에서 비동기 전환된 비율 / 초대 후 "이게 뭔 회의죠?" 문의(DM) 발생률 ↓ / 종결 카드 작성률 / "필요했음" 피드백 비율.

## 6. 열린 질문

- **Q-F1.** 초대 역할(결정권자/정보/실행)을 참석자 전원에게 공개할까, 본인에게만 표시할까? (역할 공개는 회의 내 발언 위계를 만들 수 있음)
- **Q-F2.** 발의 게이트의 강도 — 차단(비동기 강제)인가 넛지(역제안 후 통과 허용)인가? 초기엔 넛지가 안전.
- **Q-F3.** 종결 카드 미작성 시 처리 — 리마인드만? 다음 발의 시 "지난 회의 미종결" 표시? (강제성과 도구 피로의 트레이드오프)
- **Q-F4.** 정기 회의 재검증 주기 N의 기본값 (예: 8회마다 or 분기마다).
- **Q-F5.** 하이브리드 형식에서 공정성(P5) 적용 — 항상 같은 사람만 원격인 경우의 로테이션 개념 필요 여부.

## 7. 조직 맥락 보정 (v0.3) — 계기 종속 · 복수 안건 · 위계

초기 설계(§1~§2)를 실제 회사에서 회의가 구성되는 맥락에 비추어 세 가지 보정한다. 원칙: **양식이 조직을 바꾸게 하지 말고, 양식이 조직을 따라가게 한다.**

### 7.1 계기가 이후 질문을 결정한다 (계기 종속)

Q0의 계기·Q1의 목적·Q2의 안건은 독립 입력이 아니라 **종속 관계**다. "결정 필요"라는 계기를 골랐다면 목적은 이미 "결정"이고, 물어야 할 것은 "어떤 결정을 내려야 하는가"다. 계기를 고르는 순간 나머지 질문이 바뀐다:

| 계기 | 안건 질문 | 목적 기본값 (수정 가능) | 안건 형태 |
|---|---|---|---|
| 결정 필요 | "어떤 결정을 내려야 하나요?" | 결정 | 단일 중심 |
| 문제 발생 | "어떤 문제가 발생했나요?" | 결정 | 단일 중심 |
| 정보 공유 | "무엇을 공유하나요?" | 공유 → **게이트 발동** | 단일 |
| 정기 리듬 | "이번 회차에서 다룰 안건은?" | 정렬 | **복수 기본** |
| 이전 회의 후속 | "지난 회의에서 무엇이 남았나요?" | 결정 | 복수 |

→ 발의 입력 부담이 줄고(질문이 구체적이어서 답하기 쉬움), 계기-목적 불일치(계기는 "결정 필요"인데 목적은 "관계")가 구조적으로 줄어든다.

### 7.2 회의는 복수 안건 컨테이너다

실제 정기 회의 공지는 "안건 및 준비 사항 1~5" 형태다 — 회의는 하나의 목적만을 위한 것이 아니라 **여러 안건을 조속히 처리하는 자리**인 경우가 많다. 따라서:

- 안건은 1줄 문자열이 아니라 **안건 목록(1개 이상)**. 발의 관문은 동일하다 — "안건을 하나도 쓸 수 없다면 회의가 아직 필요하지 않다."
- 안건 수 × 단위 시간이 **권장 소요 시간**에 반영된다 (안건 5개짜리 30분 회의를 발의 시점에 잡아낸다).
- 초대장·보고 메시지에 안건이 **번호 목록**으로 실린다.
- (P1) 종결 카드는 안건별 결론으로 확장 — "안건 3은 다음 회차로 이월" 같은 상태가 다음 발의의 안건 후보로 환류.

### 7.3 위계 인지 — 투표는 사라지지 않는다, 직렬화된다 (v0.6 정정)

익명 투표 수렴은 **동등한 위치의 참석자** 사이에서만 자연스럽다. 본부장 같은 상급자에게 "투표하세요"는 조직 맥락에 맞지 않는다 — 실제로는 **안건과 시간안을 정리해 보고(상신)하고 승인을 받는다**.

> ⚠️ **v0.6 정정:** 초기 v0.3 설계는 상급자 포함 시 투표 단계를 통째로 보고로 **대체**했다 — 이는 동료 참석자들의 투표권까지 없애는 결함이었다(당시 Q-F8). 올바른 구조는 대체가 아니라 **직렬화**다: 동료들은 그대로 익명 투표하고, 그 득표 결과가 보고안의 순위가 된다. 상급자만 투표자 명단에서 빠질 뿐, **누구의 투표권도 사라지지 않는다.**

| 구성 | 수렴 방식 | 동작 |
|---|---|---|
| 수평 — 동료만 | 익명 투표 | 기존 그대로. 최다 득표 = 확정. 집계 수만 공개 |
| 수직 — 상급자 포함 | **익명 투표 → 보고·승인 (직렬)** | ① 동료들(상급자 제외) 익명 투표 → ② 안건 번호 목록 + **득표 순 시간안 1~3안**을 보고 → ③ 상급자가 선택/승인 → 확정 |

- **동료의 표는 보고안의 순위로 반영된다** — 1안 = 최다 득표안. 상급자는 팀의 의사가 정렬된 안을 받아 결정한다 (舊 Q-F8 해소).
- 보고안에 담기는 것은 **시간안·득표 집계·가능 인원수뿐**. 누가 어디에 투표했는지, 개인별 내 기준·불가 사유는 위로도 새지 않는다 (P1·비밀 투표 원칙은 위계 방향으로도 동일).
- 상급자의 캘린더도 free/busy로만 조율에 반영된다 — 상급자 역시 개인 축 프라이버시의 주체다.
- 상급자의 초대 역할 기본값은 **결정권자**(필참). 상급자 미승인 = 회의 미성립.
- 성립 조건의 재정의: 수평 모드 "필참 전원 수락" / 수직 모드 "상급자 승인 + 필참 수락".

### 7.4 조직도 기반 참석자 구성 (v0.5)

참석자는 자유 문자열 목록이 아니라 **조직도 엔터티**다. 사내 구성원은 이미 `본부 > 팀 > 직급 > 이름` 구조 안에 있고, 회의 편성은 그 구조 위에서 일어난다 — "막무가내로 이름을 집어넣는" 방식은 실제 조직의 인원 편성과 어긋난다.

```jsonc
Member {
  name, dept, team,                  // 본부 · 팀 (조직도/HR 연동 — DATA-SOURCING §3 "조직도/역할" 소스)
  rank: 사원|주임|대리|과장|차장|팀장|본부장,   // 직급 — 위계 판별의 근거
  duty                               // 직무 (PO, FE 리드, QA …)
}
```

- **선택 UI = 조직도 브라우징**: 본부 → 팀 그룹으로 표시, 팀 내 직급 내림차순 정렬, **팀 단위 일괄 초대/해제**. 검색은 이름·팀·본부·직급·직무 전부에 걸린다.
- **직급이 위계 분기(§7.3)의 판별 근거**: `팀장 이상 = 상급자` → 포함 시 자동으로 보고·승인 모드. 조직도에서 도출되므로 주최자가 수동 지정할 필요 없음 → **Q-F6 부분 해소** (기본: 조직도 자동, 예외적 수동 오버라이드는 열린 질문으로 유지).
- 상급자는 목록에서 직급 배지 + "보고 대상" 표시로 사전에 식별된다 — 초대하는 순간 플로우가 어떻게 바뀔지 예측 가능.
- 초대장·보고 메시지·기록의 인명 표기는 `이름 + 직급` (예: 강태호 본부장)으로 통일 — 조직의 실제 커뮤니케이션 관행을 따른다.
- **역할(§Q2)과 직급은 별개 축**이다: 직급은 조직의 사실(위계·수렴 방식 결정), 역할(결정권자/정보/실행/FYI)은 이 회의에서의 기능(필참/선택 결정). 팀장이라도 특정 회의에선 FYI일 수 있다.

**추가 열린 질문:**
- **Q-F6.** ~~상급자 판별~~ (7.4에서 부분 해소 — 조직도 직급 기준 자동) → 남은 쟁점: 팀장 임계치가 조직마다 다를 수 있음(임원만 보고 대상인 조직) — 임계 직급의 조직별 설정.
- **Q-F7.** 다단 위계(본부장 + 팀장 동시 참석) 시 승인 순서와 대리 승인(비서·부재 시) 처리.
- **Q-F8.** ~~수직 모드에서 동료들의 내 기준·투표 반영~~ (7.3 v0.6 정정으로 해소 — 투표는 유지되고 득표 순이 보고안의 순위가 됨) → 남은 쟁점: 상급자가 추천 1~3안 **밖의** 시간을 지정하는 경우의 처리 (재조율 제안 vs 그대로 수용).
- **Q-F12.** 겸직·TF 소속(한 사람이 두 팀)의 조직도 표현과 팀 일괄 초대 시 중복 처리.

### 7.5 목적은 복수다 (v0.7)

§7.1은 계기가 목적 **기본값**을 정한다고 했지만, 실제 회의의 목적은 하나가 아니다. 앞서 본 팀 미팅 메모만 해도 안건1(배포 완료 = **공유**)과 안건5(피드백 수집 방안 논의 = **결정**)가 한 자리에 섞여 있었다. 목적을 단일 선택으로 강제하면 **거짓 정밀도**가 생긴다 — 회의를 억지로 한 상자에 넣는 것.

따라서 목적 유형은 **복수 선택**이다. 계기가 정한 기본 목적 하나가 선택된 상태로 시작하되, 사용자가 필요한 만큼 더하고 뺀다. 이때 목적이 파생시키던 값들은 **집합에서 계산**한다:

> **"정보 공유와 결정이 한 번에 필요하면?"** — 계기(단일)와 목적(복수)은 다른 축이다. 계기는 "이 회의가 어디서 생겼나"(주된 발단, 하나), 목적은 "무엇을 하나"(복수)다. 답: 계기는 "결정 필요" 하나로 두고, **목적에서 결정 + 공유를 둘 다 선택**한다. 그러면 비동기 게이트는 뜨지 않고(결정이 있으니 회의가 정당), 초대장·기록에 두 목적이 모두 실린다. ⚠️ v0.8 초기 구현은 목적을 한 줄로 접어(`목적 · [결정] 유형 변경`) 이 복수 선택을 숨겨, 사용자가 계기에서 둘을 누르려 하는 혼란이 있었다 → v0.9에서 **목적을 눈에 보이는 복수 선택 칩으로 환원**(계기 종속으로 기본 하나만 켜두되 칩은 노출). 출처·구성 설정은 계속 접어 둠(그쪽이 진짜 부차적 항목).

| 파생값 | 단일 선택(구) | 복수 선택(v0.7) |
|---|---|---|
| 권장 소요 시간 | 그 목적의 권장 시간 | **선택된 목적들의 최댓값** — 섞인 회의는 가장 긴 것 이상 필요 (안건 수 × 15분과도 비교해 큰 값) |
| 비동기 게이트 | 목적 = 공유일 때 발동 | **선택이 "공유" 하나뿐일 때만 발동** — 공유+결정처럼 동기 목적이 섞이면 게이트 없음 (결정이 있으니 회의가 정당) |
| 의사결정 방식 필드 | 목적 = 결정일 때 표시 | **선택에 "결정" 포함 시 표시** |
| 초대장·기록 배지 | 배지 1개 | 선택된 목적 **여러 배지** |

> 이 변경은 게이트 로직을 오히려 **더 정확하게** 만든다. 단일 선택에서는 "공유+결정" 회의도 목적을 "결정"으로만 찍으면 게이트를 회피할 수 있었지만, 복수 선택에서는 "공유"를 정직하게 포함시켜도 "결정"이 함께 있으면 게이트가 뜨지 않는다 — 억지 회피가 필요 없어진다.

- 기록(MeetingRecord)의 목적도 배열로 저장한다. 다만 안건별 목적까지 쪼개지는 않는다 — 목적은 **회의 단위**의 집합이고, 안건별 성격은 종결 카드의 안건별 결론(§8)이 담는다. (안건별 목적 태깅은 과잉 설계로 판단 — 열린 질문 Q-F13.)

**추가 열린 질문:**
- **Q-F13.** 목적을 회의 단위 집합으로 둘지, 안건별로 태깅할지 — 현재는 회의 단위. 안건 수가 많고 성격이 크게 갈리는 회의에서 안건별 목적이 필요해지는지 관찰 후 결정.

### 7.6 UX 원칙 — 회의를 잡는 일 자체가 스트레스면 안 된다 (v0.8)

기능을 하나씩 정당화하며 쌓다 보니(계기·목적·안건·출처·역할·형식·결정방식) 정작 회의 하나 잡는 일이 **심문**이 됐다. 이는 이 서비스가 내세운 **Ask→Derive**([DATA-SOURCING §1](./DATA-SOURCING.md)) 원칙을 UI에서만 정반대로 어긴 것이다. 세 가지 원칙으로 교정한다:

**① 점진적 공개 (Progressive Disclosure) — 묻지 말고 파생하라.**
- 발의 화면은 사실상 **계기 → 안건** 둘만 전면에 둔다. 계기를 고르면 목적이 파생되어 `목적 · [정렬] 유형 변경` 한 줄로 접히고, 출처는 `+ 배경·출처 추가` 링크 뒤로 숨는다.
- 구성 화면은 **참석자를 주역**으로 올린다. 제목(안건에서 자동)·소요시간(목적에서 파생)·형식·결정방식은 기본값과 함께 `설정 · 제목 · 30분 · 대면` 한 줄로 접히고, 필요할 때만 펼친다.
- 원칙: **모든 필드를 동시에 같은 무게로 보여주지 않는다.** 기본값으로 답할 수 있는 것은 접고, 사용자가 실제로 결정해야 하는 것만 전면에.

**② 참석자 선택은 스크롤이 아니라 탐색이다 (100명 문제).**
- 자유 목록을 100줄 스크롤하지 않는다. 세 경로를 제공: **검색 우선**(이름·팀·본부·직급·직무) / **최근 함께한 사람**(다가오는 일정·기록에서 빈도순 Derive — 재초대가 흔한 케이스를 1탭으로) / **접힌 조직도 아코디언**(팀 헤더만 보이고 클릭 시 펼침, 내 팀만 기본 펼침).
- 검색 중에는 아코디언을 접고 평면 결과만. 검색이 이미 좁혀주므로.

**③ 연속성 — 모달은 6개의 벽이 아니라 하나의 흐름이다.**
- 스텝 아래에 **연속성 요약 바**를 둔다: `정기 › 정렬 › 안건 2 › 👤 4명 › 목 14:00` 처럼 회의가 만들어지는 모습이 단계를 넘어 누적 표시된다. 사용자는 자기가 무엇을 만들고 있는지 항상 본다 — 각 스텝이 단절된 폼이 아니라 한 객체를 빚는 과정이 된다.

**④ 바로 생성 — 빠른 길도 같은 회의 객체다 (v0.19).**
- `회의 바로 생성`은 사이드바 시작 카드와 우하단 FAB 스피드다이얼의 두 경로에서 진입한다. 별도 축약 기능이 아니라 기존 회의 생성 위저드의 앞단을 미리 채우는 시트다.
- 템플릿(`30분 싱크 / 1:1 / 결정 회의`)은 제목·계기·목적·소요·형식·시간 범위를 채운다. 이후 참석자 구성, 후보 산출, 발송/동의 수렴은 기존 플로우와 동일하다.
- 이 원칙을 어기면 "빠른 생성으로 만든 회의"와 "정식 생성으로 만든 회의"가 서로 다른 데이터 품질을 갖게 된다.

**⑤ 모바일과 모달 — 작은 화면은 압축이 아니라 재배치다 (v0.19).**
- 모바일 주간 캘린더는 7열을 줄이지 않고 **데이 스트립 + 하루 시간축**으로 바꾼다. 범례는 한 줄 가로 스크롤로 유지해 ON AIR/피함/비어 있음 언어를 숨기지 않는다.
- 일반 모달은 하단 시트로 열고, 새 회의 위저드는 긴 작업이므로 풀스크린을 유지한다. 작은 확인/상세 모달과 다단계 작업 화면은 같은 크기 규칙을 쓰지 않는다.
- 설정은 카드형 운영 화면으로 분리한다. 회의 제외 시간·캘린더 연동·근무시간·데이터 관리를 한 화면에서 찾되, 실제 조율 흐름을 방해하지 않는 2뎁스에 둔다.
- 우하단 FAB는 이전처럼 `회의 바로 생성 / 내 일정 추가` 두 액션을 제공한다. 상단에는 회의 생성 CTA를 두지 않는다.

> 요지: 시스템의 목적은 "회의 발산을 줄이는 것"인데, 회의 잡기 자체가 고통이면 사용자는 도구를 버리고 옛 방식(그냥 캘린더에 꽂기)으로 돌아간다. **낮은 발의 마찰**은 부가 기능이 아니라 G4(회의 감축)의 전제다.

**추가 열린 질문:**
- **Q-F14.** 스마트 기본값의 적정선 — 형식·소요시간을 데이터에서 자동 제안할 때 사용자가 "왜 이렇게 잡혔지"를 몰라 불신하지 않도록 근거를 어디까지 노출할지.
- **Q-F15.** 대규모 초대(팀 여러 개·수십 명) 시 역할 지정의 배치 처리 — 개별 칩 토글은 인원이 많으면 다시 스트레스가 된다.

### 7.7 참석의 3축 모델 — 사람이 보여야 한다 (v0.13)

이전 UI의 결함: 플로우 전반에서 **사람이 보이지 않았다.** 주최자(나)는 화면에 아예 없었고, 참석자는 작은 칩에 축약되어 "누가·왜·어떤 자격으로" 있는지 맥락이 사라졌으며, **참석의 형태**(현장인지 원격인지)는 개념 자체가 없었다. 참석을 세 축으로 정의하고, 이를 **참석 구성 보드(로스터)** 로 상시 노출한다:

| 축 | 질문 | 값 | 어디서 오나 |
|---|---|---|---|
| **① 역할** | 왜 부르나 | 결정권자 / 정보제공 / 실행담당 / 참고 | 주최자 지정, 목적이 필수 역할 요구([PURPOSE-LOGIC](./PURPOSE-LOGIC.md)) |
| **② 의무** | 와야 하나 | **필참**(성립 조건) / **선택**(무불이익) / **열람**(비참석·기록 수신) | 역할에서 파생 |
| **③ 형태** | 어떻게 오나 | **현장** / **원격** / **기록 열람** | 회의 형식이 기본값(대면→현장, 온라인→원격), **하이브리드에서만 개인별 선택** |

**참석 구성 보드 (구성 단계의 중심):**
- **주최자(나)가 첫 행** — 이름·직급·팀·주최자·참석 형태까지. 회의를 만드는 사람도 참석자다.
- 사람마다 한 행: `[아바타] 이름 직급 · 팀 · 의무 설명("필참 — 성립 조건") · 준비물(목적에서 자동) | [역할 버튼] [형태 버튼] [제거]`
- 상단 요약: `필참 4 · 선택 1 · 열람 1 · 현장 3 · 원격 2 — 필참 전원 수락 = 성립 · 회의실 정원은 현장 N명 기준`

**형태 축의 기능적 효과 (표시용이 아니다):**
- **회의실 정원 = 현장 인원만** 계산한다. 원격·열람 인원은 방 크기와 무관 — 하이브리드 5인 회의에 원격 2명이면 3인실이면 충분.
- (P1) 이동시간 제약도 현장 참석자에게만 적용 — [DATA-SOURCING §4](./DATA-SOURCING.md)의 이동 가능성 검사 대상이 형태 축으로 좁혀진다.
- 동의 확인·초대장에도 역할·형태가 따라간다: 동의 행에 "정보제공 · 필참 · 원격", 초대장에 "참석 · 필참 4 · 선택 1 · 현장 3 · 원격 2".

**레이아웃 원칙 — 모달은 스크롤하지 않는다 (v0.14):** 구성 단계를 한 줄로 쌓으면 조직도를 스크롤하는 동안 명단이 화면 밖으로 사라진다 — "누가 선택됐는지"가 보이지 않는 원인. 구성 단계는 **두 패널**로 고정한다: 왼쪽 **사람 추가**(검색·최근·조직도, 자체 스크롤) / 오른쪽 **참석 명단**(주최자+선택 인원, 항상 표시, 자체 스크롤). 모달 높이는 고정되고 각 패널만 내부 스크롤 — 고르는 곳과 고른 결과가 **동시에** 보인다. 피커에는 선택 체크·팀별 "N 선택" 카운트로 선택 상태를 중복 표시.

### 7.8 시간의 3계층 — ON AIR 상태등과 회의 할당 체계 (v0.18)

캘린더의 시간은 ON AIR 보드에서 세 상태로 읽히고, 각 상태는 회의 할당(후보 산출)에서 **다른 강도**로 작동한다:

| 상태 | 관리 주체·방식 | 할당 시 효과 | 동료에게 |
|---|---|---|---|
| **ON AIR** (연동 일정 + ON AIR 확정 회의) | 캘린더 연동이 자동 수집, ON AIR 확정은 시스템이 기록 | **차단(하드)** — 후보에서 제외 | 바쁨으로 계산 |
| **회의 제외 시간** | 본인 소유 — 특정 날짜 한 번 또는 요일·시간을 매주 반복해 직접 설정 | **차단(하드)** — 후보에서 제외 | 사유 없이 바쁨으로만 반영 |
| **우선순위 낮음** (오늘 상태 + soft 개인 일정) | 당일 컨디션·업무량·회의 피로 또는 소프트 일정 | **감점(소프트)** — 제외하지 않고 순위만 조정 | 완전 비공개 (점수로만 반영) |
| **비어 있음** | 일정도 피함 신호도 약한 파란 시간 | 후보 생성의 기본 공간 | "가능" 신호로만 집계 |
| **개인 업무 차단** (수동 추가 hard) | 본인이 직접 추가·삭제 | **차단(하드)** — 후보에서 제외 | **사유 없이** 바쁨으로만 (제목·성격 비공개) |

> **회의 제외 시간은 하드 차단, 오늘 상태는 소프트 순위.** 사용자가 강도를 추측하지 않도록 설정 화면에서 둘을 섞지 않는다.

**화면 적재 — 슬롯 인스펙터:** 주간 캘린더의 시간을 클릭하면 그 시간의 판정이 그대로 열린다: ①ON AIR 여부 ②개인 일정 차단 여부 ③회의 제외 시간 적용 여부 ④오늘 상태의 소프트 감점 ⑤비어 있음 여부. 최종 판정은 `제외 / 후순위 / 가능` 세 값으로만 표시한다.

**오늘 상태:** "오늘 회의하기 힘들다", "오늘 업무가 너무 많다" 같은 당일 맥락은 `오늘 상태`로 취급한다. 오늘 상태는 하드 차단하지 않고 소프트 가중치로 후보를 뒤로 민다. 특정 시간을 실제로 막으려면 `회의 제외 시간 > 한 번만 제외`를 사용한다.

**"내 차례"와 플로우의 연결:** 사이드바 "내 차례"는 별도 할 일 목록이 아니라 **회의 수명주기에서 내 응답을 기다리는 블로커의 파생 뷰**다. 각 항목에 단계 칩(동의→성립→확정→종결)이 붙고, 클릭하면 그 단계의 실제 화면(동의 현황·참석 응답·회의실 배정·종결 카드)이 열리며, 풀면 목록에서 사라진다.

**추가 열린 질문:**
- **Q-F16.** 원격 참석자의 형태를 본인이 바꿀 수 있어야 하나(주최자 지정 vs 본인 선택 — 역할 이의제기와 동일 패턴).
- **Q-F17.** 하이브리드에서 "항상 같은 사람만 원격"이 되는 편중 — 공정성 로테이션(P5)의 대상에 포함할지.

## 8. 회의 기록 적재 구조 (v0.4) — 다름을 같음으로

### 8.1 기록의 재정의

회의를 "2인 이상이 **동일한 안건**을 기반으로 소통하는 행위"로 정의하면, 회의의 근본 문제가 드러난다: **같은 회의에서도 참석자마다 남는 방향성이 다르다.** A는 "결정됐다"고 기억하고 B는 "논의만 했다"고 기억한다. 이 다름이 재회의·재논의·실행 누락의 뿌리다.

> **회의 기록 = 각자의 '다름'을 팀의 '같음'으로 변환하는 행위.**
> 기록은 회의록 문서가 아니라, 안건마다 "우리가 합의한 결론은 이것"이라고 못 박는 **동기화 산출물**이다.

따라서 기록의 단위는 회의가 아니라 **안건**이다 — 회의는 복수 안건 컨테이너(§7.2)이므로, 결론도 안건별로 남아야 한다. 실제 팀의 미팅 메모("안건1. Atlas — 배포 완료, 도메인 확보 필요(담당자)…")가 이미 이 구조다.

### 8.2 기록 스키마 — MeetingRecord

```jsonc
MeetingRecord {
  meetingId, title, when, format, location, purpose, origin, scribeId,
  attendees[{ userId, role }],          // 초대 역할 그대로 보존 — 재사용의 원천
  agendas[{                             // ★ 기록의 단위 = 안건
    title,
    notes[],                            // 논의 경과 (자유 서술, 담당자 멘션 포함)
    decision,                           // 이 안건의 결론 1줄 — "결정 없음"도 결론
    actions[{ owner, due, text }],      // 실행 항목
    status: "done" | "carry"            // 완료 / 이월 ★ 연속성의 키
  }]
}
```

- **`status: carry`(이월)가 연속성의 핵심이다.** 결론이 나지 않은 안건은 실패가 아니라 "다음 회의의 안건 후보"라는 상태를 갖는다.
- 종결 카드(Q5)는 이 스키마의 **입력 UI**다: 회의 직후 기록자가 안건별 결론 1줄 + 완료/이월 + 액션을 남기면 그대로 저장소에 적재된다.
- 프라이버시 정합(§4): 기록은 전부 **회의 축** 데이터 — 참석자·FYI 전원 공개. 내 기준·오늘 상태·사유는 기록에도 존재하지 않는다.

### 8.3 연속성 — 기록에서 다음 회의가 나온다

회의는 단발이 아니라 사슬이다. 기록 저장소가 있으면 "이전 회의에서 이어지는 회의"가 1클릭이 된다:

| 진입점 | 동작 | 자동 반영 |
|---|---|---|
| 기록 상세 → **"후속 회의 만들기"** | 발의(Step 0)가 후속 모드로 열림 | 계기="이전 회의 후속" · 출처="7/2 커미티 미팅 후속" · **이월 안건 자동 등록** · 참석자+역할+형식 복사 |
| 기록 상세 → **"설정 복사해 새 회의"** | 발의가 빈 상태로 열리되 구성만 복사 | 참석자+역할+형식+제목 복사 (계기·목적·안건은 새로 — 다른 주제의 회의) |
| 발의에서 계기="이전 회의 후속" 선택 | **기록 선택기** 표시 | 과거 기록 목록에서 선택 → 위 후속 모드와 동일 반영 |

- 출처(Q4·provenance)가 이 시점에 **링크**가 된다: 후속 회의의 초대장에서 "출처: 7/2 커미티 미팅"을 누르면 그 기록으로 이동 — 회의 사슬 전체가 추적 가능.
- 이월 안건이 후속 회의에서도 이월되면 그 횟수가 쌓인다 → "이 안건은 3회째 이월 중"은 **회의로 풀 수 없는 안건**이라는 신호 (비동기 전환 또는 결정권자 재지정 제안 — 환류 Phase 5와 연결).

### 8.4 위저드·화면 반영 (index.html 적용됨)

- **상단 "회의 기록" 화면**: `정확한 일시 / 회의명·장소 / 결론·안건 / 참석자` 테이블 → 클릭 시 안건별 기록 상세(노트·결정·액션·상태) 모달. 기록자·목적 분류는 조회 UI에서 제외한다.
- **기록 상세 모달**: "후속 회의 만들기 — 이월 N건" / "설정 복사해 새 회의" 버튼.
- **Step 0 후속 계기**: 계기="이전 회의 후속" 선택 시 기록 선택기 표시, 선택 즉시 이월 안건·참석자·형식 반영.
- **종결 카드 개편**: 결정 1줄 → **안건별 결론 1줄 + 완료/이월 토글**. 배포 시 기록 저장소에 적재되어 즉시 "회의 기록" 목록에 나타난다.
