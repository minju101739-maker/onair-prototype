# ON AIR 데이터 소싱 & 회의실/공간 모델 — "폼을 채우지 않는다"

> 발상의 전환: **내 기준·제약을 사용자에게 묻지 않는다. 이미 존재하는 데이터를 긁어와 시스템이 자동으로 ON AIR/피함/비어 있음 상태를 채운다.**
> 회의실 예약 시스템은 그 데이터원 중 하나이자, 기존 캘린더엔 없던 **공간(위치) 축**을 추가한다.

- 문서 버전: v0.4
- 상위 문서: [CONCEPT.md](./CONCEPT.md) · 관련: [CONTEXT-MODEL.md](./CONTEXT-MODEL.md)
- 변경 이력: v0.2 — 위치 은닉 원칙에 **N≤2 대칭 모드 예외**(위치·이동시간 동등 공개) 추가. / v0.3 — §9 정정: 신원 없는 커버리지 카운트는 표시(CONTEXT-MODEL ①과의 모순 해소). / v0.4 — ON AIR 상태 소싱 추가: 캘린더·확정 회의는 ON AIR, 내 기준·오늘 상태는 피함, 빈 구간은 파란 조율 가능 시간.

---

## 1. 핵심 전환 — Ask가 아니라 Derive

이전 문서(CONTEXT-MODEL 6장)의 최대 리스크는 "아무도 기준 폼을 안 채운다"였다. 이 리스크는 **입력 방식을 바꾸면 사라진다.**

| | Ask 방식 (폐기) | **Derive 방식 (채택)** |
|---|----------------|------------------------|
| 제약 | 사용자가 "외근/회의" 입력 | 캘린더 free/busy + **회의실 예약 기록**에서 추출 |
| 내 기준 | 사용자가 "점심 직후 싫어" 입력 | **과거 회의 수락/거절/리스케줄 패턴**에서 추론 |
| 부담 | 높음 (안 채움) | 0 (시스템이 채움) |

> 폼은 **확인(confirm)용**으로만 남는다: "데이터를 보니 당신은 금요일 오후 회의를 자주 옮기던데, 회피 규칙으로 둘까요?" — 작성이 아니라 승인.

---

## 2. 회의실 예약 = 새로운 데이터 축 (공간)

기존 캘린더 free/busy는 **시간 축만** 안다. 회의실 예약은 **"누가, 언제, 어느 지점, 어느 방에"** 를 알려준다 → 공간 정보가 생긴다.

회의실 데이터로 새로 도출되는 것:
- **위치(지점)**: 그 시간에 그 사람이 어느 지점 건물에 있(을 예정이)는가.
- **이동시간 제약**: 14:00 강남지점 회의 → 14:30 판교지점 회의는 물리적으로 불가. free/busy로는 둘 다 "바쁨"일 뿐 이동 불가를 모른다.
- **외근 추론**: 원격 지점 예약 또는 사내 예약 없음 + 캘린더 바쁨 → "외근" 신호. (외근은 캘린더상 '한가'로 보일 수 있던 사각지대였음 — CONTEXT-MODEL ②)
- **수용 인원**: 방 정원 vs 참석자 수 → 하드 제약.

---

## 3. 데이터 소스 → 무엇을 자동으로 채우나

| 데이터 소스 | 채우는 항목 | 종류 |
|-------------|-------------|------|
| 캘린더 free/busy | 시간대 바쁨/한가 → ON AIR/비어 있음 | 제약(하드) |
| ON AIR 확정 회의 | 자체 조율로 성립한 회의 블록 | ON AIR(하드) + 회의 축 |
| **회의실 예약 시스템** | 위치·이동시간·정원·외근 추론 | 제약(하드) + 지점 기준 |
| 근무시간 데이터(HR/근태) | 출근·점심·퇴근 앵커 | 내 기준 앵커 기준값 |
| **과거 회의 행동 로그** | 수락/거절/리스케줄/지각 패턴 → 시간대·요일 기준 | 피함/좋음(소프트) |
| 오늘 상태 | 컨디션·업무량·회의 피로 | 피함(당일 소프트) |
| 조직도/역할 | 필참/선택 기본 추정 | 우선순위 |

→ 이 매핑이 CONTEXT-MODEL의 3단계 퍼널(①우선순위 ②제약 ③기호)을 **사용자 입력 없이** 채운다.

## 4. "슬롯"의 재정의: (시간) → (회의실 × 시간)

회의실이 들어오면 후보 단위가 바뀐다.

```
이전 후보 = { 시간 }
이후 후보 = { 회의실(지점·방·정원) × 시간 }
```

### ② 제약(하드)에 추가되는 것
- **룸 가용성**: 그 방이 그 시간에 비어 있는가 (회의실 예약 시스템).
- **수용 인원**: 방 정원 ≥ 필참자 수.
- **이동 가능성**: 각 참석자의 직전·직후 일정 위치 ↔ 이 방 사이 이동시간이 확보되는가.

### ③ 기호(소프트)에 추가되는 것
- **지점 기준**: "내 거점 지점에서 하는 회의가 좋음"(이동 0).
- **이동 부담**: 이동시간이 길수록(타지점) 감점 — 하드 탈락은 아니지만 불편.

## 5. 전환된 워크플로우 — Posit → Evaluate

```
[주최자] "판교 3F-회의실B, 목 14:00, 30분" 이라고 가정해서 던짐
        │
        ▼
[시스템] 각 참석자의 개인 평가기가 데이터로 판정 (사유는 본인 경계 안에서만):
   - C: 직전 13:30 강남지점 예약 → 이동시간 부족  → 불가(하드)
   - A: 데이터상 목요일 오후 좋음                → 점수 좋음(소프트)
   - D: 정원 초과 없음, 거점 일치                → 좋음
        │
        ▼
[결과] "이 조합은 4/5 가능. 1명 이동시간 부족."
        + 대안: "같은 시간 강남지점 방으로 바꾸면 5/5" 자동 역제안
```

- 주최자가 **구체적 회의를 던지면**, 시스템이 가부와 더 나은 대안을 돌려준다.
- 동시에 시스템이 **상위 (회의실×시간) 조합을 먼저 추천**할 수도 있다 (두 방향은 같은 엔진의 양면).

## 6. 프라이버시 재조정 — 이게 가장 중요하다 ⚠️

"데이터를 긁어온다"는 강력하지만, **동료 비열람 원칙(CONCEPT P1)** 과 정면으로 부딪힐 수 있다. 두 가지 긴장:

### 긴장 1 — 프로파일링의 역설
행동 로그로 "쟤는 금요일 오후 싫어함"을 추론하는 건, 본인이 말한 적 없는 기준을 시스템이 만들어내는 것 = 더 침습적일 수 있다.
- **해소:** 추론된 내 기준은 **본인 경계(개인 평가기) 안에서만** 생성·저장되고, **본인에게만** 표시·확인된다. 동료·주최자·중앙 서버로 절대 나가지 않는다. (위임-집계 모델 그대로)

### 긴장 2 — "누가 막았나"의 노출
주최자가 "판교 14시"를 던졌을 때, "1명 불가"까지는 괜찮지만 **"C가 막았다 / C가 이동 안 됨"** 이 보이면 사회적 압박 + 위치 노출이 된다.
- **설계 노브(열린 질문):**
  - 소프트(기호) 레이어 → **신원 없이 집계 점수/카운트만** 노출.
  - 하드(제약) 레이어 → 필참자가 진짜 불가하면 신원 필요할 수도. 어디까지 익명화할지 결정 필요.
- 위치 정보는 특히 민감 → 동료에게는 "이동시간 부족"의 **사유(어느 지점인지)** 를 숨기고 가부만.

> 원칙 재확인: 주최자에게 가는 것은 **조합별 종합 점수 / 가능 인원수 + 대안 제안**. "왜, 누가"는 경계 밖으로 안 나간다.

> **예외 — N≤2 대칭 모드:** 위 위치 은닉은 **N≥3 익명 집계 모드**에 한한다. 참석자 2명 이하의 대칭적 상호 공개 모드([CONCEPT 4.2](./CONCEPT.md))에서는 **위치·지점·이동시간도 양쪽이 동등하게 공개**한다 — 어차피 둘만의 회의는 상호 협상이고, 한쪽만 위치를 가리는 건 비대칭이라 오히려 불공정하기 때문. 단 여기서도 노출은 `what`(어느 지점/이동 가부)까지이고, 내 기준·오늘 상태 등 `why`는 개인 평가기를 안 넘는다.

## 7. 콜드스타트 & 추론 신뢰도

데이터 기반의 약점도 정직하게:

- **콜드스타트**: 신입·신규 도입 시 행동 로그가 없음. → 근무시간 앵커(HR 데이터)로 **기본 기준**부터 시작하고, 로그가 쌓이며 정교화.
- **거짓 패턴**: "금요일 회의 3번 옮김"이 기준이 아니라 우연일 수 있음. → **신뢰도(confidence)** 를 함께 계산하고, 임계치 넘을 때만 본인에게 "피함 기준으로 둘까요?" 확인. 낮은 신뢰도는 약한 가중치로만.
- **데이터 접근 동의**: 행동 로그·근태·회의실 기록을 긁는 것 자체에 사전 동의·범위 고지 필요(거버넌스).

## 8. 가정 회의실 예약 시스템 — "RoomBook" (가정 스펙)

실제 연동 시스템은 미정이므로, 아래 가정 모델 위에서 설계를 진행한다. 추후 실제 시스템에 맞춰 어댑터만 교체.

```jsonc
Branch  { id, name, geo }                         // 강남, 판교, 분당 …
Room    { id, branchId, name, floor, capacity, equipment[] }
Booking { id, roomId, start, end, holderId }      // 점유자 식별만; 우리는 title/참석자 디테일 안 읽음
```
가정 API:
```
GET /branches
GET /rooms?branchId=
GET /bookings?roomId=&from=&to=     // 룸 free/busy (점유 구간 + holderId)
```
- **읽기 범위**: 룸 점유 구간 + 점유자 ID까지만. 회의 제목·참석자 명단·사유는 **읽지 않는다**(캘린더와 동일한 최소 노출 원칙).
- **점유자 ID 용도**: 오직 *그 사용자 본인의 개인 평가기*가 자기 이동시간을 계산하는 데만 쓰임. 타인에게 위치가 노출되지 않음.
- **이동시간(Q2 가정)**: 1차로는 **지점 간 고정 매트릭스**(예: 강남↔판교 40분, 동일건물 층간 5분). 추후 지도 API로 고도화.

## 9. Q3 정책 — "누가 막았나"는 노출하지 않는다 (확정)

선택지 붕괴 모델([CONTEXT-MODEL 2.1](./CONTEXT-MODEL.md))에 따라, 주최자에게 가는 것은 **살아남은 확실한 3~5개 슬롯뿐**이다.
- 하드·소프트 모두 **신원·사유 비노출**. 단 **신원 없는 커버리지 카운트(예: 4/6명 가능)는 표시한다** — 이는 CONTEXT-MODEL ①의 커버리지 등급화(슬롯을 죽이지 않고 밝히기)에 필수이며, 숨기는 것은 "누가/왜"이지 "몇 명"이 아니다. (v0.3 정정 — 이전 문구 "카운트조차 숨김"은 CONTEXT-MODEL 3-①의 커버리지 히트맵과 모순이었음)
- 필참자가 전원 불가해 후보가 0이 되는 예외 상황에서만, "탐색 범위를 넓힐까요" 식의 **익명 안내**로 처리.
