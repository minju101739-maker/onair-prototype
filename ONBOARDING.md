# ONBOARDING.md — 제품에서 회수한 설명글과 온보딩 설계 원천

- 문서 버전: v1.2
- 개정: 2026-07-12

> **원칙 (2026-07-09 확정):** 제품 UI에는 메커니즘 설명·프라이버시 안내·유도 문구를 넣지 않는다.
> 그 역할은 1회성 온보딩이 전담한다. UI에 허용되는 텍스트는 ①데이터 사실(건수·시간·장소)
> ②판정 결과 문장("이 시간은 추천하지 않아요") ③빈 상태의 짧은 사실 진술("지금 답할 초대가 없어요")뿐이다.
>
> 아래는 UI에서 **삭제(회수)한 카피의 전량 보관소**이자, 온보딩이 가르쳐야 할 개념 목록이다.
> 문구는 삭제 당시 원문 그대로 보존한다 — 온보딩 화면에서 재사용하거나 다듬어 쓴다.

---

## 1. 회수된 카피 전량 (화면별)

### 1.1 개인 축 프라이버시 (가장 중요한 온보딩 주제)

| 있던 곳 | 회수된 원문 |
|---|---|
| 슬롯 인스펙터("이 시간은") 하단 | 🛡 판단 근거는 나만 봐요 — 동료에겐 가능 여부만 전해져요 |
| 초대 모달(동의 요청) 하단 | 🛡 불가 사유·내 캘린더 상세는 주최자에게 공개되지 않아요 — 전달되는 것은 동의/불가뿐 |
| 초대 모달(참석 여부) 하단 | 🛡 불참 사유·내 캘린더 상세는 주최자에게 공개되지 않아요 — 전달되는 것은 참석 여부뿐 |
| 일정 상세 메타 꼬리표 | · 나만 봐요 |
| 일정 추가 프리뷰 꼬리 문장 | 동료에겐 사유 없이 결과만 전해져요. |
| 캘린더 범례(내 기준 옆) | 🛡 나만 봄 |

**가르칠 개념:** 개인 축(불가 사유·개인 일정 상세·선호 점수)은 어떤 화면에서도 동료에게 공개되지 않는다.
동료에게 전달되는 것은 가능/불가(동의/불가·참석 여부)라는 **결과**뿐이다.

### 1.2 조율 규칙 (성립 조건)

| 있던 곳 | 회수된 원문 |
|---|---|
| 위저드 참석자 단계 · 명단 푸터(선택 시) | **필참**만 성립 조건에 들어갑니다. 선택은 가능하면 참석, 공유만은 참석 없이 초대·기록만 받습니다. |
| 위저드 참석자 단계 · 명단 푸터(기본) | 필참 전원 동의 = 성립 |
| 위저드 참석자 단계 · 명단 요약 우측 노트 | 필참 전원 동의 = 성립 |

**가르칠 개념:** 참석 자격 3단계(필참/선택/공유만)와 성립 조건 — 필참 전원 동의(+상급자 있으면 승인) 후에만 확정.

### 1.3 응답(내 차례) 개념

| 있던 곳 | 회수된 원문 |
|---|---|
| 좌측 응답 패널 부제 | 내가 답해야 상대의 조율이 진행돼요 |
| 통계 카드 "응답할 초대" 서브 | 답변을 기다리고 있어요 / 지금 답할 초대가 없어요(카드용) |
| 응답 목록 모달 빈 상태 꼬리 | 새 초대가 오면 캘린더와 여기에 떠요. |

**가르칠 개념:** 답할 초대가 있으면 좌측에 패널이 나타나고, 다 답하면 사라진다. 내 응답이 상대 조율의 블로커다.

### 1.4 조율 파이프라인 (보낸 조율·회의 후 정리)

| 있던 곳 | 회수된 원문 |
|---|---|
| 통계 카드 "보낸 조율" 서브 | 동의를 모으고 있어요 / 회의 잡기로 시작해 보세요 |
| 통계 카드 "회의 후 정리" 서브 | 결론을 기록으로 남겨주세요 / 정리할 회의가 없어요 |
| 보낸 조율 목록 모달 빈 상태 꼬리 | — 회의 잡기로 시작해 보세요. |

**가르칠 개념:** 발의 → 동의 수집 → 확정 → 종결 카드(기록)로 이어지는 수명주기. 통계 카드가 각 단계의 진입점.

### 1.5 회의 만들기 흐름

| 있던 곳 | 회수된 원문 |
|---|---|
| 바로 생성 모달 히어로 | 템플릿과 시간만 먼저 정하고, 참석자와 후보 검증은 이어서 진행합니다. |
| 바로 생성 모달 히어로(구성원 선택 진입 시) | 템플릿과 시간만 정하면 선택한 구성원이 참석 명단에 미리 담깁니다. |
| 바로 생성 프리뷰 설명 | 참석자 단계에서 필참/선택/공유만을 정하면 ON AIR가 가능한 파란 시간을 계산합니다. |
| 바로 생성 프리뷰 인원 꼬리 | — 참석 명단에 미리 담겨요 |
| 위저드 빈 참석 명단 | (아직 나뿐입니다) — 검색·최근·조직도에서 사람을 담으세요 |
| 위저드 사람 담기 힌트 | 회사 N명 · **검색 우선** |

**가르칠 개념:** 두 진입점(바로 생성=시간 먼저 / 전체 위저드=목적 먼저), 참석자는 검색·최근·조직도 3경로로 담는다.

### 1.6 구성원 조회

| 있던 곳 | 회수된 원문 |
|---|---|
| 선택 바(0명일 때) | 구성원을 선택하면 바로 회의를 구성할 수 있어요 |
| 선택 바(0명일 때, 버튼 옆) | 행을 눌러 선택하세요 |
| 검색 입력 placeholder | 검색어 입력 후 Enter 또는 조회 |
| 빈 상태(하위부서 꺼짐) 꼬리 | 하위부서포함을 켜거나 왼쪽에서 본부·팀을 선택하세요. |
| 빈 상태(검색 0건) 꼬리 | — 검색조건·검색어를 바꿔 다시 조회해 보세요. |

**가르칠 개념:** 리스트에서 체크로 선택 → 선택 바 CTA로 즉시 회의 구성. 선택은 페이지·검색·조직 범위를 넘어 유지된다.

### 1.7 캘린더 색 시스템 (텍스트 라벨을 색으로 대체 — 라벨 원문도 보관)

UI에서 삭제된 라벨: 존의 `피함/좋음 + 사유(집중·휴식·점심 후·업무량·칼퇴)`, 슬롯의 `우선순위 낮음`,
개인 일정 칩의 `밀림/차단`, 참석자 아바타, 초대 블록의 `동의 필요/참석 여부` 칩.

**가르칠 색 문법 (2026-07-09 확정: 내 기준은 막음/우선순위 낮음 2단계 — 좋음(가점)은 폐기):**
- **사건(회의)**: 분류색 soft 배경 — 팀 정기 teal · 1:1/리뷰 violet · 외부/채용 amber / ON AIR 확정 = blue
- **응답 필요** = 흰 바탕 + 무채색 대시 테두리(아직 자리만 잡힘) + `응답` 배지
- **우선순위 낮음**(내 기준·근무시간 밖) = 채도를 낮춘 연회색 워시 — 사유는 hover·슬롯 클릭에서
- **막음** = 회색 채움 블록(막아둔 시간·개인 차단) — 이 시간엔 회의가 잡히지 않음
- **상태**: 지난 날 흐림 · 개인 = 흰 바탕 실선

### 1.8 `ON AIR 확정`의 의미 (2026-07-12 교정)

`ON AIR 확정`은 현재 회의가 실시간으로 진행 중이라는 뜻이 아니다. **ON AIR에서 참석자·시간을 조율하고 확정한 회의**라는 출처와 상태를 뜻한다. 캘린더 카드·범례·목록 배지는 모두 프라이머리 블루로 표시한다.

**가르칠 개념:** 파란색 `ON AIR 확정` = ON AIR 조율을 마친 확정 회의. 현재 진행 상태나 단순 업무 일정에는 이 표현을 사용하지 않는다.

## 2. 온보딩 체계 v1.2 (2026-07-12 — 허브 + 화면 안내 + 시나리오 가이드 4종)

처음 진입하면 **화면 안내**(코치마크 8스텝)가 자동 시작되고(localStorage `onairTour`),
상단 **? 아이콘 = 온보딩 허브** — 화면 안내와 시나리오 가이드 전체 목록을 연다.

### 2.1 화면 안내 (코치마크 8스텝)

실제 화면 요소를 스포트라이트로 비추고 `짧은 제목 + 지금 할 행동 한 문장`으로 안내한다.
FAB → 응답 패널 → 수명주기 4카드 → 캘린더 색 4판정 → 회의 제외 토글 → 구성원 탭 → 기록 탭 → ? 아이콘.

모든 스텝은 `이전 / 다음 / 건너뛰기(자동 안내) 또는 가이드 종료(수동 안내)`를 제공한다. 첫 스텝의 이전 버튼만 비활성화한다. 키보드는 `← / → / Esc`로 이전·다음·종료를 실행한다.

### 2.2 시나리오 가이드 — 실제 UI를 그대로 구동하는 워크스루

시나리오 스텝은 강조된 요소를 직접 누르거나 `다음`으로 같은 동작을 실행한다(`step.do`). 이전을 누르면 시작 화면으로 복귀한 뒤 목표 스텝 직전까지 필요한 동작을 다시 재생한다.

| 가이드 | 스텝 | 흐름 (전부 실제 UI 구동) |
|---|---|---|
| **회의 잡기** | 9 | + 버튼 → 새 회의 만들기 → 목적(결정 필요) 선택 → 안건 입력 → 참석자 단계 → 구성원 검색·추가 → 참석 자격 구분 → 시간 단계 → 추천 시간 확인(발송 전 종료) |
| **내 시간 지키기** | 6 | + 버튼 → 내 일정 추가 → 막기/우선순위 2단계 선택 → 매주 반복 탭 → 반복 일정 관리 목록 → 캘린더 빗금·연회색 판정으로 복귀 |
| **초대에 답하기** | 3 | 응답 패널 → 동의/불가 이분법·사유 비공개·전원 동의=확정 → 응답 전 대시 카드의 의미 |
| **회의 정리** | 4 | `정리할 회의` 카드 → 목록에서 회의 진입 → 안건별 결론 입력 → 저장 결과와 공유 범위 확인(가이드에서는 저장하지 않음) |

각 시나리오는 종료·건너뛰기 시 `cleanup`으로 열린 모달을 원상 복구한다. 가이드는 실제 초대 발송이나 회의 기록 저장을 수행하지 않는다.

구현: `TOURS{intro,create,protect,respond,wrap}` / `startTour(key)` / `renderTour`(#tourlayer — 앱 모달과 분리된 레이어라
위저드가 열린 채로 그 위에 코치마크가 뜬다). 스포트라이트 = 어둠막 4장 + 링(구멍 안 요소는 실제 클릭 가능),
대상 요소는 `tour-preview`에 시각 복제한다. 화면 밖 대상은 먼저 중앙으로 스크롤하고, 말풍선은 아래·위·오른쪽·왼쪽 중 대상과 겹치지 않는 위치를 선택한다. 조건부 대상이 누락되면 관계없는 UI를 대신 강조하지 않고 전체 어둠막 위 중앙 안내를 유지한다. `rebuildTourTo(index)`가 이전 단계의 화면 상태를 재구성한다.

### 2.3 행동 문장 규칙

- 제목은 대상이나 작업 이름만 적는다.
- 본문은 사용자가 지금 할 행동 하나를 65자 이하로 적는다.
- 문장은 `누르세요`, `선택하세요`, `확인하세요`, `입력하세요`처럼 긍정형 지시로 끝낸다.
- 다음 단계의 의미, 정책 설명, 예외 조건은 현재 코치마크에 함께 넣지 않는다.
- `TOUR_COPY_BANNED`와 본문 길이 검사가 모든 `TOURS` 문구를 확인한다.

## 3. 운영 규칙

- UI에 새 설명글이 필요해 보이면 **여기에 적재하고 온보딩 스텝에 배정**한다. 제품 화면에는 넣지 않는다.
- 검증 하네스에 "설명글 부재" 어서션이 있다(priv-line·rs-sub·pl-reason 부재 검사) — UI 재유입 시 실패한다.
